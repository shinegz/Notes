# LLM Wiki：让知识自己长出来

> RAG 每次都在重新发现知识。LLM Wiki 把知识编译一次，然后持续累积。

## 30 秒心智模型

**核心问题：知识管理系统的知识是消耗品还是资产？**

传统 RAG 把文档当仓库——你存进去，LLM 用时取出来，每次都在重新理解。LLM Wiki 把知识当代码——你写一次，LLM 维护它，交叉引用自动生成，知识随时间增值。

**三层架构：**

```
┌─────────────────────────────────────────────────────────┐
│  Schema (CLAUDE.md / AGENTS.md)                        │
│  定义 LLM 如何维护 Wiki 的规则                           │
├─────────────────────────────────────────────────────────┤
│  Wiki (markdown 文件)                                   │
│  LLM 生成和维护的知识库，交叉链接已建立                  │
├─────────────────────────────────────────────────────────┤
│  Raw Sources (原文文档)                                 │
│  不可变的事实来源，LLM 只读不写                          │
└─────────────────────────────────────────────────────────┘
```

---

## RAG 的困境

RAG（Retrieval-Augmented Generation）的工作模式是：上传文档 → LLM 检索相关片段 → 生成回答。问题在于，每次提问，LLM 都要从零开始理解和组装答案。

举例来说，如果你有 50 篇论文，想问"这几篇论文对某个问题的看法有什么演变"，RAG 只能把相关段落拼在一起。LLM Wiki 则会维护一个持续更新的观点页面，每次新论文进来自动更新这个页面的内容。

RAG 是知识的临时工，干完就走。LLM Wiki 是知识的建筑师，一直在工地上。

## 核心操作

LLM Wiki 有三个核心操作：

### 1. Ingest（摄入）

把新文档丢进 Raw Sources，告诉 LLM 处理。LLM 会：

- 读取文档，提取关键信息
- 写一个摘要页面到 Wiki
- 更新相关的实体页面和概念页面
- 在 index.md 中登记
- 在 log.md 中记录这笔操作

一篇文档可能触发 10-15 个 Wiki 页面的更新。

### 2. Query（查询）

你向 Wiki 提问，LLM 搜索相关页面，综合回答并注明来源。关键是：**好的回答本身应该沉淀回 Wiki**。你问了一个对比分析，下次再问类似问题，直接读沉淀的页面就行。

### 3. Lint（整理）

定期让 LLM 做健康检查：

- 页面间是否有矛盾
- 是否有被新来源更新的过时断言
- 是否有孤岛页面（没有入链）
- 是否有重要概念只被提及但没有专属页面
- 交叉链接是否完整

## 两个关键文件

### index.md —— 内容目录

每个 Wiki 页面一条记录，包含链接、一句话摘要、日期。LLM 回答问题时先读 index，找到相关页面再深入。在 ~100 个来源、几百个页面的规模下，这个文件本身就是搜索引擎。

### log.md —— 操作日志

按时间顺序记录所有操作：摄入、查询、整理。格式统一（如 `## [2026-04-02] ingest | Article Title`），可以用 `grep "^## \[" log.md | tail -5` 快速查看最近动态。

## 适用场景

| 场景 | 怎么用 |
|------|--------|
| **个人成长** | 追踪目标、健康、心理——日记、文章、播客笔记，日积月累形成自我画像 |
| **学术研究** | 花几周或几个月深入一个主题——读论文、读报告，建一个随研究进化的综合 Wiki |
| **读书** | 每读一章建一个页面，建人物、主题、情节的页面，读完就有了一个丰富的伴侣 Wiki |
| **团队知识** | 内部 Wiki 由 LLM 维护，来源是 Slack、会议记录、项目文档 |
| **竞争分析 / 做课 / 深度爱好** | 任何需要持续积累知识的场景 |

---

## 为什么有效

维护知识库最累的不是读和想，而是**归档、交叉引用、保持一致**。这些事情人类懒得做，但 LLM 擅长。LLM Wiki 把这些脏活累活外包给 LLM，你负责两件事：**找来源、问好问题**。

LLM 写，你读。像结对编程：你盯着屏幕提要求，LLM 在另一边改文件。

---

## 与传统 PKM 的区别

| 维度 | 传统 PKM（Notion/Obsidian） | LLM Wiki |
|------|---------------------------|----------|
|谁来维护 | 你手动维护 | LLM 自动维护 |
| 知识增长 | 依赖你的主动整理 | 每条信息自动连接 |
| 交叉引用 | 靠你自己想起来 | LLM 主动建立 |
| 矛盾处理 | 靠你自己发现 | LLM 标记并提示 |

## 快速上手

```
1. 建三个目录：
   - /raw-sources/    # 放原始文档
   - /wiki/           # LLM 生成的知识库
   - /schema.md       # 告诉 LLM 如何维护 Wiki

2. 放第一篇文档到 raw-sources/

3. 告诉 LLM：
   "请处理这个文档，生成 Wiki 页面，更新 index 和 log。"

4. 之后每次加新文档，都重复第 3 步。
```

## 工具推荐

- **Obsidian** —— Wiki 的 IDE，图形视图看整体结构
- **Obsidian Web Clipper** —— 浏览器插件，把网页文章一键转 markdown
- **qmd** —— 本地 Markdown 搜索引擎，BM25/向量混合搜索，支持 CLI 和 MCP
- **Marp** —— 用 Markdown 写幻灯片，Obsidian 有插件
- **Dataview** —— Obsidian 插件，用 SQL 风格查询页面元数据
