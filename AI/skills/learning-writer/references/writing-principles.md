## 去 AI 味完整指南

### 核心原则速查

处理文本时牢记这 5 条原则：

1. **删除填充短语** - 去除开场白和强调性拐杖词
2. **打破公式结构** - 避免二元对比、戏剧性分段、修辞性设置
3. **变化节奏** - 混合句子长度。两项优于三项。段落结尾要多样化
4. **信任读者** - 直接陈述事实，跳过软化、辩解和手把手引导
5. **删除金句** - 如果听起来像可引用的语句，重写它

## 30 秒心智模型的写法

**核心原则：心智模型要准确传达问题本质，不是为了类比而类比。**

| 主题类型 | 推荐心智模型方式 | 原因 |
|---------|-----------------|------|
| 技术/算法原理 | **问题-解决方案框架** | 直接切入痛点，展示解决思路 |
| 概念/理论 | 问题-解决方案或精确定义 | 概念需要准确理解，类比可能误导 |
| 历史/演进 | 时间线或关键转折点 | 演进脉络比类比更重要 |
| 对比/决策 | 核心差异表格 | 直接展示差异更清晰 |

**问题-解决方案框架模板：**

```
## 30 秒心智模型

**核心问题：[一句话描述问题]**

[问题现象的简要说明]

**[技术/方案] 的解决方案：[一句话描述核心思路]**

[简要展示关键变化，如代码对比、结构对比]

**关键变化：**
1. [变化点1]
2. [变化点2]
3. [变化点3]
```

**何时可以用类比？**
- 概念高度抽象，需要借助熟悉事物理解
- 读者是初学者，需要降低认知门槛
- 类比域与概念有清晰、准确的映射关系

**何时不该用类比？**
- 类比只能部分对应，容易误导读者
- 技术细节需要精确理解
- 问题本身可以用代码/结构直接说明
- 读者已经有一定基础，不需要弯路

**类比的自检标准：**
- 每个类比元素在原概念中有明确对应
- 不会因为类比而产生错误理解
- 删掉类比后，读者仍能理解核心
- **类比必须解释 WHY**：不仅要描述机制是什么，还要解释为什么这样设计

**类比失败的常见原因：**

| 失败模式 | 问题 | 修正方向 |
|---------|------|---------|
| 只描述 WHAT | "事件委托就像总邮箱分发邮件" | 解释为什么：统一管理带来什么好处 |
| 类比过度延伸 | 用一个类比套所有子概念 | 每个子概念可能需要独立的解释 |
| 隐喻映射不清 | 类比元素与原概念对应模糊 | 明确列出映射关系 |
| 忽略设计动机 | 只说是什么，不说为什么这样设计 | 回到问题-解决方案框架 |

**利用名称特征的技巧：**

很多技术概念的名称本身就包含了核心隐喻，善用这些名称特征比强行类比更有效：

| 概念名称 | 名称蕴含的隐喻 | 有效的利用方式 |
|---------|--------------|---------------|
| React | "反应式" - 响应变化 | "UI 是状态的镜像，状态变，UI 跟着变" |
| Hook | "钩子" - 钩住某物 | "钩子钩进 React 内部，把状态拉出来用" |
| Fiber | "纤维" - 细丝 | "渲染被拆成细小的纤维，可以一根根处理" |
| Memo | "备忘录" - 记下来 | "把上次结果记下来，下次直接用" |
| Virtual DOM | "虚拟的" | "用 JS 对象模拟真实 DOM，在内存里先算好" |
| Event Delegation | "委托" | "事件处理委托给祖先节点统一管理" |

**示例对比：**

```
// ❌ 弱：强行类比，不利用名称
Hooks 让函数组件可以拥有状态，就像给函数加上了记忆功能...

// ✅ 强：利用名称特征
Hooks 的 "hook" 就是钩子。函数组件本来是纯函数，没有状态。
Hook 钩进 React 内部的状态系统，把状态拉出来给函数用。
钩几个就有几个状态，钩的位置不能变，所以不能在条件里调用。
```

```
// ❌ 弱：只描述 WHAT，没有 WHY
React 把所有事件集中到根节点处理，就像公司有一个总邮箱分发给员工。

// ✅ 强：利用名称特征 + 解释 WHY
Event Delegation 的 "delegation" 就是委托。
为什么要委托？
1. 省内存：不用给每个元素绑事件监听器
2. 好管理：动态添加的元素自动有事件处理
3. 跨浏览器：React 统一处理兼容问题
委托给谁？
React 17 之前委托给 document，之后委托给 root 容器。
```

## 深度解析类文章的必要章节

**对于涉及内部原理的主题（如框架、算法、系统），必须包含：**

### 1. 数据结构详解

- 核心数据如何存储和组织
- 用 ASCII 或表格展示结构
- 解释每个字段的作用

```
示例：

### Hook 对象结构

┌─────────────────────────────────────┐
│           Hook 对象                  │
├─────────────────────────────────────┤
│ memoizedState  : any     // 状态值   │
│ baseState      : any     // 初始状态 │
│ queue          : Object  // 更新队列 │
│ next           : Hook    // 下一个   │
└─────────────────────────────────────┘
```

### 2. 核心算法流程

- 关键操作的执行流程
- 用流程图或伪代码展示
- 解释为什么这样设计

### 3. 关键代码解析

- 实际实现片段（简化但准确）
- 不要只贴代码，要逐行解释
- 展示关键边界条件处理

```javascript
// 示例：关键代码要逐行解释

function useState(initialState) {
  // 1. 获取或创建 Hook 节点
  const hook = mountWorkInProgressHook();
  
  // 2. Mount 阶段：使用初始值
  if (isMount) {
    hook.memoizedState = initialState;
  }
  
  // 3. Update 阶段：处理更新队列
  else {
    hook.memoizedState = processUpdateQueue(hook);
  }
  
  // 4. 返回状态和 dispatcher
  return [hook.memoizedState, dispatch];
}
```

### 4. 设计权衡

- 为什么选这种实现方式
- 对比其他可行方案
- 解释取舍理由

**深度解析的自检清单：**

- [ ] 读者能否根据数据结构描述，画出结构图？
- [ ] 读者能否根据算法描述，写出简化版实现？
- [ ] 关键代码是否覆盖了核心路径？
- [ ] 是否解释了「为什么这样设计」而非只说「是什么」？

## 核心原则

### 1. 结论先行

- 开篇 30 秒内给出核心结论
- 用一句话概括全文主旨
- 不要铺垫，直接上干货

**差：**
> 在深入探讨之前，我们需要先了解一些背景知识...

**好：**
> React Fiber 让渲染可中断，解决了大型应用卡顿问题。

### 2. 一图胜千言

- 复杂概念必须可视化
- 每个图表只讲一个核心概念
- 图表后加「读图结论」点明要点

### 3. 简单、精准、犀利

- 删除所有冗余词汇
- 用具体数字替代模糊描述
- 一句话能说清，不用两句

**差：**
> 此外，值得注意的是，这个系统的设计在很大程度上体现了其对于可扩展性的高度重视。

**好：**
> 系统支持水平扩展，单集群可处理 100万 QPS。

### 4. 展示概念层级关系

- 明确区分核心概念与从属概念
- 用层级结构展示依赖关系
- 让读者一眼看出重点

### 5. 分类例子的使用原则

**何时用分类表格：**
- 多个并列项有共同属性，方便横向对比
- 读者需要快速了解每种类型的核心特点
- 分类本身是理解主题的关键

**何时不用分类表格：**
- 项目间关系复杂，不能简单并列
- 需要解释每个项目"为什么"而不仅是"是什么"
- 分类会掩盖真正重要的差异

**示例对比：**

```
// ✅ 适合分类：事件传播阶段
| 阶段 | 方向 | 用途 |
|------|-----|------|
| 捕获 | 外→内 | 预处理、拦截 |
| 目标 | 到达目标 | 处理事件 |
| 冒泡 | 内→外 | 委托处理 |

// ❌ 不适合分类：性能优化工具
| 工具 | 作用 | 
|------|------|
| React.memo | 阻止重渲染 |
| useMemo | 缓存计算 |
| useCallback | 缓存函数 |

// ✅ 更好的方式：问题驱动
问题：父组件渲染，子组件跟着渲染
解决：React.memo 阻断渲染

问题：计算成本高，每次都算
解决：useMemo 缓存结果

问题：函数作为 props，每次都是新引用
解决：useCallback 保持引用稳定
```

### 6. 易混概念的澄清原则

**识别易混概念：**
- 同一领域内名称相似的概念
- 经常一起出现但职责不同的概念
- 对新手来说容易张冠李戴的概念

**澄清方法：**

```
1. **并排对比**
   | 概念 A | 概念 B |
   |-------|-------|
   | 是什么 | 是什么 |
   | 做什么 | 做什么 |
   | 何时用 | 何时用 |

2. **层级关系图**
   概念 A (上层)
      └── 概念 B (实现层)
      └── 概念 C (实现层)

3. **演进关系**
   旧方案 → 解决问题 X → 新方案
   新方案继承了 Y，但改变了 Z
```

**常见易混类型：**

| 类型 | 示例 | 澄清方式 |
|------|------|---------|
| 算法 vs 实现 | Reconciliation vs Fiber | Reconciliation 是算法（做什么），Fiber 是实现（怎么做）|
| 接口 vs 实现 | Hook API vs Hook 内部 | API 是承诺的接口，内部是实现细节 |
| 概念 vs 工具 | 并发模式 vs useTransition | 并发模式是特性，useTransition 是使用入口 |
| 相似名称 | useMemo vs useCallback | useMemo 缓存值，useCallback 缓存函数 |
| 相邻阶段 | 渲染阶段 vs 提交阶段 | 渲染可中断，提交不可中断 |

**必须澄清的场景：**
- 文章涉及多个相关概念时，开篇先建立概念地图
- 概念间存在包含/交叉关系时，用图示说明
- 历史演进导致概念重叠时，说明演变过程

## 语言表达

### 删除这些 AI 痕迹

| AI 味 | 改为人话 |
|-------|---------|
| 此外/值得注意的是 | 直接删掉 |
| 至关重要/核心关键 | 说明为什么重要 |
| 深入探讨/详细分析 | 直接讲内容 |
| 综上所述/总而言之 | 直接说结论 |
| 不仅...而且... | 拆成两句或只留重点 |
| 从...到...（虚假范围） | 具体列举 |

### 变化节奏

- 长短句交替使用
- 偶尔用短句制造停顿
- 避免连续三个句子结构相同

**示例：**
```
长句：Fiber 架构通过将渲染工作拆分为可中断的小单元，让浏览器有机会处理用户输入，从而解决了 React 在大型应用中的卡顿问题。

短句：这是关键。

中句：每个 Fiber 节点代表一个工作单元，可以被暂停、恢复或丢弃。
```

### 注入观点

- 不要只是中立陈述
- 表达真实看法（"我觉得..." / "问题在于..."）
- 承认复杂性（"这看起来矛盾，但..."）

## AI 写作模式检测与修正

### 模式 1：过度强调意义

**危险词汇：** 作为/充当、标志着、见证了、是……的体现/证明、极其重要的/至关重要的/核心的/关键性的作用、凸显/强调了其重要性、反映了更广泛的、象征着、为……做出贡献、为……奠定基础、关键转折点、不断演变的格局、不可磨灭的印记、深深植根于

**改写前：**
