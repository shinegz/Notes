# Test Best Practices 技能介绍

本文档与 `SKILL.md` 保持同步，是面向人类的中文导航说明。

---

## 语言要求

本 skill 的**所有输出内容必须使用中文**，包括：测试计划、测试代码注释（JSDoc、行内注释）、`describe`/`it` 名称的描述部分、E2E 脚本注释、E2E 测试报告、与用户的所有对话回复。

代码本身（函数名、变量名、API 调用）保持英文，注释和说明文字统一用中文。

---

## 它是什么

**Test Best Practices** 是一套通用测试技能：把自然语言描述的需求（功能、界面行为、验收标准）或粘贴的代码，转化为**专业的测试策略**与**可落地的测试代码**。

核心理念：**测试的终极目标是还原用户真实操作流程，确保每个业务场景都被可靠验证**。

---

## 何时会用到它

在以下情况应触发此技能：

- 描述功能、界面、验收标准（AC），需要配套测试或验证思路
- 明确要求编写测试、用例、补覆盖率或验证行为
- 粘贴代码并问「这该怎么测」
- 意图明显是**验证行为**或**质量保障**，即使未出现「测试」二字

## 工作流程总览

```
用户输入（AC / 功能 / UI / 代码）
        │
        ▼
Phase 0: 上下文探测
   ├── 读取 package.json、配置文件、已有测试
   ├── 识别技术栈、测试框架、项目约定
   └── 信息不足时向用户简短追问
        │
        ▼
Phase 1: 测试计划（供你审阅）
   ├── 解析输入 → 结构化「测试对象」
   ├── 强制三层扫描（单元/集成/E2E，不可跳过）
   ├── 决定测试类型与粒度
   ├── 输出结构化测试计划（中文）
   └── 请你确认后再进入下一阶段
        │
        ▼
Phase 2A: 单元测试实现 → 【关卡确认】
   ├── 按模块拆分测试文件（一个文件 = 一个功能模块）
   ├── 写入代码、执行验证
   └── 输出完成确认才能进入 2B
        │
        ▼
Phase 2B: 集成测试实现 → 【关卡确认】
   ├── 分析模块协作机会
   ├── 有则实现，无则说明理由
   └── 输出完成确认才能进入 2C
        │
        ▼
Phase 2C: E2E 测试实现 → 【关卡确认】
   ├── 【必须】输出 Mock 数据清单
   ├── 写脚本 + agent-browser 实跑
   ├── 截图、报告
   └── 禁止归咎于账号问题
        │
        ▼
Phase 3: 持久化与可追溯
   ├── 保存所有产物到仓库
   ├── 语言验证（全中文输出）
   └── 确认所有产物可被人类和 Agent 读懂
```

**⚠️ 关键改进**：Phase 2 分为三个子阶段，每个子阶段有**关卡确认**，确保不会遗漏任何测试类型。

## 核心哲学

测试的终极目标是还原用户真实操作流程，确保每个业务场景都被可靠验证。

---

## 测试金字塔：策略的核心

```
          ╱╲
         ╱E2E╲       ← 10%：昂贵、慢
        ╱──────╲
      ╱Integration╲    ← 20%：中等成本
     ╱──────────────╲
    ╱   Unit Tests   ╲  ← 70%：便宜、快
```

**核心原则**：大量单元测试 + 适量集成测试 + 少量 E2E 测试。一个 E2E 测试的成本可能是单元测试的 10 倍以上，细节验证交给低层测试，E2E 只覆盖关键用户旅程。

---

## 测试文件组织原则

### 单元测试 & 集成测试：Co-location

测试文件必须与被测源码放在同一目录。

### E2E 测试：集中目录

E2E 测试采用集中目录 `tests/e2e/`（含 scripts、screenshots、reports）。

---

### ⚠️ 反模式警示

**测试的目的是验证「代码是否按预期工作」，而非「测试用例是否通过」。** 适用于所有测试类型。

判断标准：如果被测的真实代码有 bug，这个测试会失败吗？
- 会 → ✅ 有效测试
- 不会（因为 mock 返回的是正确的值）→ ❌ 无效测试

**Mock 边界原则**：只在系统边界 mock，单元测试可以 mock 模块边界，但集成测试和 E2E 必须让真实代码执行。

---

## 通用测试原则

### 黄金法则：设计瘦测试

测试代码与生产代码不同，要使它变得**极其简单、短小、没有抽象、扁平化**。一段测试代码需要做到让人一眼就能看出其目的。

### AAA 模式

每个测试分为三段：**准备（Arrange）、执行（Act）、断言（Assert）**。

### 三元命名法

测试名称必须包含三部分：被测什么、在什么条件下、期望什么结果。

### 黑盒测试

**只测 public 方法，不测内部实现细节。**

> 以上原则适用于所有测试类型（单元、集成、E2E）。

---

## Phase 0：上下文探测

在分析需求之前，先理解项目环境，保证生成的测试与仓库习惯一致。

**自动探测包括：**

1. **包清单**：`package.json`、`requirements.txt` 等，识别测试框架及脚本
2. **配置文件**：`jest.config.*`、`playwright.config.*` 等，了解路径别名与自定义行为
3. **已有测试**：抽样阅读学习命名、目录、mock 方式
4. **源码结构**：决定测试文件放置位置

**仅在无法推断时**再简短追问用户。

**冲突处理**：

当探测到的项目规范与 Skill 中的最佳实践不一致时，**必须询问用户以哪个为准**：

```markdown
检测到项目约定与最佳实践存在差异：
- 项目约定：[具体内容]
- 最佳实践：[具体内容]

请问以哪个为准？
1. 遵循项目约定
2. 采用最佳实践
```

## Phase 1：测试计划

### 输入的三种形态

| 形态 | 处理方式 |
|------|----------|
| **验收标准（AC）** | 抽取 Given/When/Then；补充隐含的错误态、边界与空态 |
| **功能 / UI 描述** | 拆成离散行为，推断隐含 AC，必要时先与用户对齐 |
| **代码 / 组件** | 分析公开 API、分支与错误处理、需 mock 的依赖 |

### 强制三层扫描（不可跳过）

在输出计划之前，必须对需求做完整的三层扫描，每一层都主动寻找测试机会。**不能因为已找到单元测试机会就停止扫描。**

| 层级 | 扫描问题 | 典型信号 |
|------|---------|----------|
| **第一层：单元测试** | 哪些逻辑是纯函数或无副作用的计算？ | 数据提取/映射函数、枚举判断、边界值计算 |
| **第二层：集成测试** | 哪些业务规则需要多个模块协同才能验证？ | 模块间数据流、状态+副作用组合、条件分支依赖多方状态 |
| **第三层：E2E 测试** | 哪些行为必须在真实浏览器中才能验证？ | 完整用户流程、视觉与布局状态、跨页面状态保持 |

> **E2E 用例数量原则**：分支逻辑属于第一/二层，E2E 只验证最终 UI 渲染结果。每条用户旅程写一条 E2E，不为每种用户状态分别写一条。所有用户状态通过 mock 控制，不依赖特定测试账号。

如果某一层为空，必须在计划中**明确说明理由**，而不是静默忽略。

### 测试类型决策

| 信号 | 推荐类型 |
|------|----------|
| 纯函数、工具、数据变换 | 单元测试 |
| 带 props/state 的组件 | 单元 + 组件测试 |
| Hook 协同、跨模块数据流 | 集成测试 |
| API 端点、服务方法 | 单元 + 集成测试 |
| 跨页面的用户流、UI 交互 | E2E |
| 校验规则、表单 | 单元 + 集成 |
| 第三方集成 | 集成（mock 边界） |

**未经确认，不进入 Phase 2。**

## Phase 2：测试实现

### ⚠️ Phase 2 关卡机制

Phase 2 分为 **2A（单元）→ 2B（集成）→ 2C（E2E）** 三个子阶段：

- 每个子阶段完成后**必须输出确认**才能进入下一个
- 禁止跳过任何子阶段
- 如果某阶段为空，必须**明确说明理由**

### Phase 2A：单元测试

**核心规则：一个测试文件 = 一个功能模块**，禁止把不同模块的测试混在一个文件中。

#### 文件规范

- **文件头 JSDoc**：每个单测文件必须包含功能归属、迭代信息、业务规则
- **命名规则**：`describe`/`it` 使用中文，描述业务场景

#### 单元测试报告

**必须输出单元测试报告**：Phase 2A 完成后，必须生成并保存单元测试报告。

**报告位置**：`tests/unit/reports/<feature-slug>-unit-<YYYYMMDD>.md`

#### 完成确认

单元测试全部完成后，输出确认：
- 测试文件列表
- 文件组织检查
- 执行结果
- 测试报告路径

---

### Phase 2B：集成测试

**⚠️ 禁止跳过**：即使 Phase 2A 完成，也必须检查是否有集成测试机会。

#### 集成测试识别检查点

| 检查点 | 对应集成测试 |
|--------|--------------|
| 多个模块协作完成一个业务行为 | 模块协作测试 |
| API 调用后数据在组件间流转 | 数据流测试 |
| 用户操作触发多个状态更新 | 状态联动测试 |
| 条件渲染依赖多个数据源 | 条件组合测试 |

#### 集成测试核心原则

**Mock 边界原则**：集成测试只在系统边界（API/网络）mock，不在模块边界 mock，让真实代码执行。

#### 完成确认

集成测试完成后输出确认，或明确说明跳过理由。

#### 集成测试报告

**必须输出集成测试报告**：Phase 2B 完成后，必须生成并保存集成测试报告。

**报告位置**：`tests/integration/reports/<feature-slug>-integration-<YYYYMMDD>.md`

---

### Phase 2C：E2E 测试

#### ⚠️ E2E 前置条件：Mock 数据清单

**禁止在没有 Mock 数据清单的情况下开始 E2E 执行。**

在执行任何 E2E 测试之前，必须先输出 Mock 数据清单：

| 场景 | API 返回值 | localStorage 状态 | 权限/开关 |
|------|-----------|-------------------|-----------|
| 场景1 | `GET /api/xxx → {...}` | `key: value` | `flag: true` |

#### ⚠️ Mock 数据迭代循环（强制要求）

**当 E2E 执行发现 mock 数据不对或不足时，必须重新生成 mock 方案并重试，而非直接标记 BLOCKED。**

迭代流程：
```
E2E 执行 → 遇到阻塞 → 分析原因
                        │
                        ├─ Mock 缺失/错误 ──► 更新 Mock 清单 ──► 重试
                        │
                        └─ 其他原因 ──► 按阻塞处理流程处理
```

**强制要求**：
- 每个阻塞必须至少尝试 1 次重新生成 mock 并重试
- BLOCKED 场景必须记录「已尝试的 mock 方案」

#### 执行流程

1. 输出 Mock 数据清单（必须）
2. 确认 dev server 运行
3. 写 E2E 脚本文件
4. 通过 agent-browser 逐步执行
5. **每个关键断言点必须截图**
6. 执行完毕写报告

**阻塞处理**：禁止归咎于账号问题，必须先检查 mock 数据。

## Phase 3：持久化与可追溯

### 必须保存到仓库的产物

| 产物 | 位置 | 用途 |
|------|------|------|
| 单元测试文件 | 与源码并列（`.test.ts`） | 规格说明 + 回归 |
| 单元测试报告 | `tests/unit/reports/<功能名>-unit-<YYYYMMDD>.md` | 执行结果归档 |
| 集成测试文件 | 与源码并列或 `__tests__/integration/` | 协作验证 |
| 集成测试报告 | `tests/integration/reports/<功能名>-integration-<YYYYMMDD>.md` | 协作验证结果 |
| E2E 脚本文件 | `tests/e2e/scripts/<功能名>.sh` | 可重复执行 |
| E2E 截图 | `tests/e2e/screenshots/<YYYYMMDD>/` | 视觉证据 |
| E2E 测试报告 | `tests/e2e/reports/<功能名>-e2e-<YYYYMMDD>.md` | PASS/FAIL 历史 |

### 语言验证点

**Phase 3 结束时，必须验证所有输出为中文**：

- [ ] 测试计划：中文
- [ ] 单元测试：`describe`/`it` 名称中文，注释中文
- [ ] 单元测试报告：中文
- [ ] 集成测试：同上
- [ ] 集成测试报告：中文
- [ ] E2E 报告：中文
- [ ] 与用户对话：中文
