# 大语言模型架构与训练流程

> 从海量文本中学会"鹦鹉学舌"到"理解意图"，大模型经历了什么。

## 30 秒心智模型

**大语言模型的训练是一个"先学语言、再学做事"的过程。**

预训练阶段：喂给它整个互联网的文本，让它学会预测下一个词——这让模型"会说人话"。

后训练阶段：教它遵循指令、写出有用的回答——这让模型"会做人话"。

两阶段缺一不可。没有预训练，模型连完整句子都写不出来；没有后训练，模型只会续写文本，不知道用户想要什么。

## 1. GPT 系列架构演进

### GPT-1 到 GPT-4 的脉络

```
GPT 系列演进

GPT-1 (2018)
├── 参数量: 1.17 亿
├── 创新: 首次证明预训练+微调范式有效
└── 架构: 12 层 Transformer 解码器

GPT-2 (2019)
├── 参数量: 15 亿
├── 创新: Zero-shot 能力，Prompt 工程的开端
└── 争议: OpenAI 因为安全考虑延迟发布

GPT-3 (2020)
├── 参数量: 1750 亿
├── 创新: Few-shot In-context Learning
│         (不给示例能推理，给几个示例能模仿)
└── 成本: 约 460 万美元训练费用

GPT-4 (2023)
├── 参数量: 未公开 (估计 1-1.8 万亿)
├── 创新: 多模态输入、复杂推理、更长上下文
└── 突破: 通过人类考试、编程竞赛
```

### 架构变体

| 架构 | 代表模型 | 特点 | 擅长 |
|------|----------|------|------|
| **Decoder-only** | GPT 系列、LLaMA | 单向自回归 | 文本生成 |
| **Encoder-only** | BERT | 双向理解 | 分类、NER |
| **Encoder-Decoder** | T5、BART | 完整结构 | 翻译、摘要 |

大模型时代，Decoder-only 成了主流。原因：

1. 训练目标统一（下一个词预测）
2. 工程优化成熟
3. 生成能力更强

## 2. 预训练：学会语言本身

### 下一个词预测

预训练的核心任务是语言建模（Language Modeling）：

```
给一个序列 [w₁, w₂, ..., wₜ]，预测 wₜ₊₁

示例：
输入: "今天天气真"
目标: "好"

输入: "机器学习是"
目标: "一种"

输入: "我想和你一起"
目标: "去"
```

这个任务看似简单，但解决它的模型必须理解：

- 语法结构
- 语义关系
- 世界知识
- 上下文推理

因为预测"好"需要知道"天气真__"是个常见搭配，预测"去"需要知道"一起__"后面常跟地点。

### 训练数据

```
预训练数据来源与占比（典型大模型）

    ┌────────────────────────────────────────┐
    │  Common Crawl (60-70%)                │
    │  网页文本，规模最大但噪声多             │
    ├────────────────────────────────────────┤
    │  Wikipedia (5-10%)                    │
    │  百科全书，知识密集且准确              │
    ├────────────────────────────────────────┤
    │  Books (10-20%)                       │
    │  长文本，叙事和推理能力强              │
    ├────────────────────────────────────────┤
    │  GitHub (5-10%)                      │
    │  代码，提升编程和结构化推理             │
    ├────────────────────────────────────────┤
    │  ArXiv (2-5%)                        │
    │  学术论文，提升科学推理                │
    └────────────────────────────────────────┘
```

### 数据处理流程

原始网页文本不能直接喂给模型。需要经过：

```
数据处理 pipeline

Raw Text (PB 级别)
      │
      ▼
[1] 质量过滤
      │
      ├──► 规则过滤：去广告、去 HTML
      ├──► 机器学习过滤：训练分类器识别高质量内容
      └──► 语言识别：保留目标语言
      │
      ▼
[2] 去重
      ├──► 精确去重：完全相同的段落
      └──► 近似去重：Near-duplicate 检测
      │
      ▼
[3] 安全过滤
      ├──► 敏感内容检测
      └──► 隐私信息删除
      │
      ▼
[4] Tokenization
      ├──► BPE / WordPiece / SentencePiece
      └──► 将文本切分为模型认识的 token
      │
      ▼
Clean Tokens (~1-3T tokens)
```

### Tokenization：让机器理解文字

Token 是模型处理的最小单元。大模型通常使用子词（Subword）分词法：

```
Tokenization 示例

句子: "人工智能正在改变世界"

BPE分词:
"人工" / "智能" / "正在" / "改变" / "世界"
= 5 tokens

英文对比:
"TheTransformer" → "The" / "Transformer"
```

中英文的 token 效率差异很大——中文平均 1.5 个字符一个 token，而英文单词可能需要 1-3 个 token。

### 训练目标详解

下一个词预测的数学表达：

```
给定上下文 x₁, x₂, ..., xₜ

最大化: Σ log P(xₜ₊₁ | x₁, ..., xₜ; θ)

或者等价地，最小化交叉熵损失:
L = - Σ log P(xₜ₊₁ | x₁, ..., xₜ; θ)
```

训练时，将整个语料库拼接成一个超长序列，用滑动窗口遍历：

```
训练样本构建

语料: [文档1] [文档2] [文档3] ...

滑动窗口 (窗口大小 = 2048):

样本1: [ 文档1 的前 2048 tokens ] → 预测下一个
样本2: [ 文档1 的第 2-2049 tokens ] → 预测下一个
样本3: [ 文档1 的第 3-2050 tokens ] → 预测下一个
...     (移动 1 token)

跨文档处理:
文档1 结束后，从文档2 开头继续
中间没有 [SEP] 等分隔符——模型必须自己学会分文档
```

## 3. 后训练：从文本续写到指令遵循

预训练完成后，模型是一个"天才鹦鹉"——它能写出流畅的句子，但不知道你想要什么。

后训练（Post-training）让模型从"续写大师"变成"有用助手"。

### 监督微调 (SFT)

用人工标注的问答数据直接训练模型：

```
SFT 数据示例

{
  "instruction": "把下面的句子翻译成英文",
  "input": "今天天气真好",
  "output": "The weather is really nice today."
}

{
  "instruction": "写一首关于春天的诗",
  "input": "",
  "output": "春风拂面柳丝长，\n桃花绽放映日光。\n..."
}
```

```
SFT 训练方式

与预训练相同的目标：下一个词预测
区别在于数据格式：
- 预训练：直接拼接文档
- SFT：拼接 [指令] + [回答]

训练时对 [指令] 部分不计算 loss（称为 User Prefix）
只对 [回答] 部分计算 loss
```

### 为什么不能直接用 SFT？

SFT 有几个问题：

1. **标注成本高**：高质量的指令-回答对需要人工撰写
2. **分布窄**：只能覆盖标注者想到的场景
3. **模型偏好**：模型会模仿标注者的写作风格和偏好

RLHF 在 SFT 基础上进一步优化，解决这些问题。

## 4. RLHF：让 AI 更像人

RLHF（Reinforcement Learning from Human Feedback）三步走：

```
RLHF 流程

                    人类偏好
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Step 1: SFT                                           │
│  用标注数据微调预训练模型 ──► SFT Model                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Step 2: 训练 Reward Model                             │
│                                                         │
│  对同一指令，模型生成多个回答                           │
│  人类标注哪个更好                                       │
│  训练 Reward Model 学习人类偏好                         │
│                                                         │
│  prompt₁ ──► response₁ (+1) ─┐                         │
│          ──► response₂ (-1)  ├─► RM 学会给"好回答"高奖赏 │
│          ──► response₃ (0)    │                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Step 3: RL 优化 (PPO)                                 │
│                                                         │
│  用 Reward Model 提供的信号，通过强化学习优化 SFT 模型  │
│  最大化累积奖励（同时加入 KL 散度约束防止跑偏）          │
│                                                         │
│  Prompt → SFT Model → Response ──► Reward               │
│                          ↑                    │         │
│                          │                    ▼         │
│                          └── KL(π || π_sft) ← 惩罚      │
│                                                         │
└─────────────────────────────────────────────────────────┘
                       │
                       ▼
                 RLHF Model (最终模型)
```

### Reward Model 的训练

Reward Model 本质上是个打分模型：

```
Reward Model 架构

输入: [prompt] + [response]
      │
      ▼
   SFT Model (冻结，只替换最后层)
      │
      ▼
   输出: 单一标量分数

训练目标：
最大化被选中的回答的分数，最小化被拒绝的回答的分数

L = -E[log σ(r_chosen - r_rejected)]
```

### PPO 优化

PPO（Proximal Policy Optimization）是一种稳定的策略优化算法：

```
PPO 目标函数

L = E[min(r_t · A_t, clip(r_t, 1-ε, 1+ε) · A_t)] - c · KL(π_θ || π_ref)

其中:
r_t = 新策略 / 旧策略 的概率比
A_t = 优势函数（基于 Reward Model）
clip = 裁剪，防止更新过大
KL = KL 散度惩罚，防止策略变化太大
c   = KL 惩罚系数
```

KL 惩罚项很关键——没有它，模型会过度优化 Reward，走向"作弊"而不是真正提升质量。

### DPO：绕过 Reward Model

2023 年底，DPO（Direct Preference Optimization）提出更简单的方法：

```
DPO vs PPO

PPO: 需要训练 Reward Model + PPO 优化
     三个模型：SFT、RM、PPO Policy

DPO: 只需要偏好数据
     直接优化策略，无需 Reward Model

DPO 损失函数:
L = -E[(x, y_w, y_l) ~ D] [log σ(β · (log π(y_w|x) - log π(y_l|x)) - β · (log π_ref(y_w|x) - log π_ref(y_l|x)))]

本质：用对比学习的思想，直接让模型偏好更好的回答
```

## 5. 训练基础设施与成本

### 分布式训练

单个 GPU 的显存放不下千亿参数。训练需要多卡并行：

```
张量并行 (Tensor Parallelism)
每个 Transformer 层在多个 GPU 间切分

Layer N
  │
  ├──► GPU 0: Attention Q, K, V 计算
  ├──► GPU 1: Attention 输出投影
  ├──► GPU 2: FFN 第一层
  └──► GPU 3: FFN 第二层

all-reduce 同步中间结果
```

```
数据并行 (Data Parallelism)
不同 GPU 处理不同数据批次

GPU 0: batch_1 ──► forward ──► loss ──► backward ──► grad
GPU 1: batch_2 ──► forward ──► loss ──► backward ──► grad
GPU 2: batch_3 ──► forward ──► loss ──► backward ──► grad
GPU 3: batch_4 ──► forward ──► loss ──► backward ──► grad
                          │
                          ▼
                    All-Reduce Gradients
                          │
                          ▼
                    Optimizer Update
```

现代大模型训练通常结合多种并行策略：**3D Parallelism** = 数据并行 × 张量并行 × 流水线并行。

### 显存与内存墙

```
大模型训练的内存消耗

以 LLaMA-65B (650 亿参数) 为例：

参数量:          650 亿
存储 (FP32):     260 GB
梯度 (FP32):     260 GB
优化器状态 (FP32): 520 GB   ← 最大头！Adam 需要一阶和二阶动量
────────────────────────────
总 FP32:         ~1040 GB  ≈ 1 TB

实际常用 FP16/FP32 混合:
总: ~800 GB

单卡 A100 (80GB): 需要 10+ 张卡才能装下
```

### 训练成本估算

```
主流大模型训练成本

模型          参数量    GPU数量    训练时间    总算力(FP16)
LLaMA-7B      7B        64         21天        1.3M GPU小时
LLaMA-65B     65B       2048       20天        6.3M GPU小时
GPT-3 175B    175B      1024+      ~30天       3.6M+ GPU小时
GPT-4         ~1T       数千        数月        (未公开)

按 AWS A100 价格 ($2/GBPU小时)：
LLaMA-65B ≈ $150,000

```

## 6. Scaling Laws：规模的力量

### Kaplan 等人的发现 (2020)

OpenAI 发现：模型的性能（困惑度）与模型规模、数据规模、计算量呈幂律关系。

```
Scaling Laws

Loss ∝ N^(-0.076) · D^(-0.095) · C^(-0.050)

N = 模型参数量
D = 训练 token 数
C = 计算量 (FLOPs)

含义：
- 模型参数量翻倍，loss 下降约 5%
- 训练数据量翻倍，loss 下降约 7%
- 计算量翻倍，loss 下降约 3.5%
```

### Chinchilla 的修正 (2022)

DeepMind 发现了一个问题：Kaplan 的实验把模型大小和数据量严重不平衡——大模型用相对少的数据训练。

```
Chinchilla 建议

原来 Kaplan 的做法：
GPT-3 175B  → 300B tokens
计算量固定 → 模型变大，数据不变

Chinchilla 的建议：
同等计算量下，模型参数量和训练 tokens 应该同比放大

LLaMA 实践：
7B  → 1T tokens
13B → 1T tokens
33B → 1T tokens
65B → 1.4T tokens

结论：计算预算应该 50/50 分配给模型和数据
```

### 涌现能力

当模型规模超过某个阈值后，会突然出现小模型不具备的能力：

```
涌现能力示例

BIG-Bench 任务性能 vs 模型规模

                    GPT-3
                       │
             PaLM 540B ├─
                       │    ╭── 三位数加法
                       │   ╱
任务                   │  ╱   ─── 笑话理解
性能                   │ ╱
(超出随机)             ├─
                       │╱
                       │
                       └─────────────────────────► 模型规模
                       10M    100M    1B     10B   100B

某些能力在小模型上几乎为零，大到临界点后突然出现
这让 AI 安全和研究变得更加不可预测
```
