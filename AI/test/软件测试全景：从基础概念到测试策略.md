## 30 秒心智模型

把测试体系想象成**医院分诊系统**：

| 概念 | 医院类比 | 职责 |
|------|----------|------|
| 单元测试 | 门诊化验 | 针对单个"细胞"（函数/模块）做最基础的检查，快速、低成本 |
| 集成测试 | 专科会诊 | 检查多个"器官"（模块）是否协同工作正常 |
| 端到端测试 | 全身体检 | 模拟真实患者（用户）流程，验证整体健康状态 |
| 测试金字塔 | 分诊制度 | 大量基础检查（单元测试）作为底座，少量昂贵检查（E2E）作为顶层 |
| TDD | 预防医学 | 先定义"什么是健康"，再进行治疗（写代码） |
| 测试替身 | 标准病人 | 用模拟对象代替真实器官，隔离测试环境 |

**一句话总结**：测试是用系统化的方式验证代码"是否按预期工作"，测试金字塔告诉我们"用什么比例做这件事"，TDD 告诉我们"什么时候做这件事"。

---

## 核心结论

1. **测试金字塔是黄金比例**：70% 单元测试 + 20% 集成测试 + 10% E2E 测试——大量快速反馈的测试在底层，少量昂贵但关键的测试在顶层。
2. **100% 代码覆盖率 ≠ 无 Bug**：覆盖率是量化指标，只告诉你"测试走了哪些路"，不告诉你"测试走得对不对"。有意义断言比覆盖率数字更重要。
3. **回归与维护是长期成本**：发布与重构依赖稳定的自动化回归；测试代码与产品代码一样需要评审与重构，避免脆性测试拖慢迭代。

## 测试是什么：边界澄清

### ❌ 测试不是什么 vs ✅ 测试是什么

| 维度 | ❌ 不是什么 | ✅ 是什么 |
|------|------------|----------|
| 本质 | 证明软件"完美无缺" | 发现软件的问题和边界 |
| 目标 | 追求 100% 覆盖率数字 | 建立对代码行为的信心 |
| 范围 | 只测"正常路径" | 测正常路径 + 边界条件 + 异常情况 |
| 时机 | 写完代码后再补测试 | 可以先写测试再写代码（TDD） |
| 角色 | 只由 QA 负责 | 开发者也要写（尤其是单元测试） |

## 测试的分类体系

### 按测试层级分类

```
┌─────────────────────────────────────┐
│          Acceptance Testing         │ ← 验收测试：满足业务需求吗？
│         (业务视角，用户参与)          │
├─────────────────────────────────────┤
│         End-to-End Testing          │ ← 端到端测试：完整流程通吗？
│         (用户行为模拟，最接近真实)     │
├─────────────────────────────────────┤
│         Integration Testing         │ ← 集成测试：模块间通信正常吗？
│           (模块协同，接口契约)        │
├─────────────────────────────────────┤
│           Unit Testing              │ ← 单元测试：这个函数对吗？
│         (最小单元，隔离测试)          │
└─────────────────────────────────────┘
```

#### 单元测试 — "门诊化验"

**定义**：测试最小可测试单元（通常是函数或方法）的正确性。

**特点**：
- 隔离性强：不依赖外部系统（数据库、网络、文件系统）
- 速度快：毫秒级执行
- 定位准：失败时能快速定位到具体代码行

**示例**：
```javascript
// 被测函数
function add(a, b) {
  return a + b;
}

// 单元测试
describe('add function', () => {
  it('should add two positive numbers correctly', () => {
    expect(add(2, 3)).toBe(5);
  });
  
  it('should handle negative numbers', () => {
    expect(add(-1, 1)).toBe(0);
  });
  
  it('should return NaN for invalid inputs', () => {
    expect(add('a', 1)).toBeNaN();
  });
});
```

**最佳实践**：
- 测试公共接口，不测私有实现
- 覆盖正常路径 + 边界条件 + 异常情况
- 一个测试只验证一个行为

#### 集成测试 — "专科会诊"

**定义**：验证多个模块或服务协同工作的正确性。

**特点**：
- 关注接口契约：数据能否正确传递
- 可能涉及外部依赖：数据库、消息队列、第三方 API
- 执行时间较长：秒级

**示例场景**：
- API 服务与数据库的交互
- 微服务之间的通信
- 前端组件与后端 API 的集成

```javascript
// 集成测试示例：测试 API 与数据库的交互
describe('User API Integration', () => {
  beforeAll(async () => {
    await database.connect('test_db');
  });
  
  afterAll(async () => {
    await database.disconnect();
  });
  
  it('should create and retrieve user', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: 'Alice', email: 'alice@example.com' });
    
    expect(response.status).toBe(201);
    
    const getUser = await request(app)
      .get(`/api/users/${response.body.id}`);
    
    expect(getUser.body.name).toBe('Alice');
  });
});
```

#### 端到端测试 — "全身体检"

**定义**：模拟真实用户操作流程，验证完整系统行为。

**特点**：
- 最接近用户真实体验
- 涉及完整技术栈：前端 → 后端 → 数据库 → 第三方服务
- 执行最慢：分钟级
- 维护成本最高：UI 变化时需要更新

**示例场景**：
- 用户登录 → 浏览商品 → 加入购物车 → 结算支付
- 表单提交 → 数据验证 → 后端处理 → 结果展示

> **对你的启发**：E2E 测试价值高但成本高，应聚焦关键用户路径，而非覆盖所有细节。细节交给单元测试和集成测试。

#### 验收测试 — "出院审批"

**定义**：验证系统是否满足业务需求和用户期望的正式测试。

**特点**：
- 从业务视角而非技术视角设计
- 通常由产品经理、业务方或最终用户参与
- 是发布前的最后一道关卡

### 按测试方法分类

#### 黑盒测试

**定义**：不关心内部实现，只基于需求规格说明进行测试。

**特点**：
- 测试者不需要了解代码结构
- 关注输入输出是否符合预期
- 适合功能测试、用户体验测试

**技术**：
- 等价类划分
- 边界值分析
- 决策表测试

#### 白盒测试

**定义**：基于代码内部结构和逻辑进行测试。

**特点**：
- 需要了解代码实现细节
- 关注代码覆盖率、路径覆盖
- 适合单元测试、安全测试

**技术**：
- 语句覆盖
- 分支覆盖
- 路径覆盖

#### 灰盒测试

**定义**：结合黑盒和白盒，既关注输入输出，又了解部分内部结构。

**特点**：
- 适合集成测试
- 可以更有针对性地设计测试用例
- 平衡测试深度和成本

| 维度 | 黑盒测试 | 灰盒测试 | 白盒测试 |
|------|----------|----------|----------|
| 内部知识 | 无 | 部分 | 完全 |
| 粒度 | 低 | 中 | 高 |
| 执行者 | 用户/QA | QA/开发者 | 开发者 |
| 适用场景 | 功能/业务测试 | 集成测试 | 单元/结构测试 |

## 测试金字塔：策略的核心

### 金字塔的起源与含义

测试金字塔由 **Mike Cohn** 在 2009 年出版的《Succeeding with Agile》中提出，是测试策略的核心模型。

```
          ╱╲
         ╱  ╲
        ╱ E2E╲        ← 10%：少量、昂贵、慢
       ╱──────╲
      ╱Integration╲   ← 20%：适量、中等成本
     ╱──────────────╲
    ╱   Unit Tests   ╲  ← 70%：大量、便宜、快
   ╱────────────────────╲
```

### 为什么是金字塔而不是倒金字塔？

| 维度 | 单元测试 | 集成测试 | E2E 测试 |
|------|----------|----------|----------|
| 执行速度 | 毫秒级 | 秒级 | 分钟级 |
| 编写成本 | 低 | 中 | 高 |
| 维护成本 | 低 | 中 | 高 |
| 问题定位 | 精准 | 中等 | 困难 |
| 反馈周期 | 极快 | 较快 | 慢 |
| 置信度 | 中 | 较高 | 最高 |

**核心原则**：
- **大量单元测试**：快速发现问题，精准定位
- **适量集成测试**：验证关键接口和协作
- **少量 E2E 测试**：覆盖核心业务流程

### 现代测试金字塔的演变

随着技术发展，测试金字塔也在演变：

```
        ╱╲
       ╱UI╲
      ╱────╲
     ╱Contract╲      ← 契约测试：API 契约验证
    ╱──────────╲
   ╱Integration ╲
  ╱──────────────╲
 ╱Unit + Component╲
╱──────────────────╲
```

部分团队还在底层增加了**静态分析**层（Lint、类型检查），构成更完整的基础。

> **对你的启发**：不要盲目追求 E2E 测试覆盖率。一个 E2E 测试的成本可能是单元测试的 10 倍以上。把钱花在刀刃上。

## TDD：测试驱动开发

### 起源与定义

TDD 由 **Kent Beck** 在 1990 年代末作为极限编程（Extreme Programming）的一部分提出。

### Red-Green-Refactor 循环

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   ┌─────────┐      ┌─────────┐      ┌──────────┐    │
│   │   RED   │ ───▶ │  GREEN  │ ───▶ │ REFACTOR │    │
│   │写失败测试│      │写通过代码│      │  优化代码  │    │
│   └─────────┘      └─────────┘      └──────────┘    │
│        ▲                                    │        │
│        └────────────────────────────────────┘        │
│                   重复循环                           │
└──────────────────────────────────────────────────────┘
```

1. **Red（红）**：写一个失败的测试，定义你想要的功能
2. **Green（绿）**：写最少的代码让测试通过
3. **Refactor（重构）**：优化代码结构，保持测试通过

### TDD 的价值

**先写测试的好处**：
1. **强制思考接口设计**：让你在写实现前先想清楚"怎么用"
2. **自然产生自测试代码**：只有通过测试才能提交代码
3. **提供即时反馈**：每个小功能都有测试覆盖
4. **重构的安全网**：有测试在，大胆重构

### TDD 的常见误区

| 误区 | 正确理解 |
|------|----------|
| TDD 能消除所有 Bug | TDD 减少缺陷，但不能替代代码审查和集成测试 |
| 必须严格遵守 Red-Green-Refactor | 实践中可以灵活调整，核心是"测试先行"的思维 |
| TDD 只适用于单元测试 | TDD 也适用于集成测试、功能测试 |

> **对你的启发**：TDD 最被忽视的步骤是 **Refactor**。很多人停在 Green 就结束，导致代码逐渐腐化。重构是保持代码健康的关键。

## 测试替身：隔离的艺术

### 为什么需要测试替身？

测试需要**隔离**——只测试目标代码，不受外部依赖影响。测试替身就是用来"替代"真实依赖的模拟对象。

这个概念由 **Gerard Meszaros** 在《xUnit Test Patterns》中系统化提出。

### 五种测试替身

| 类型 | 定义 | 特点 | 典型用途 |
|------|------|------|----------|
| **Dummy** | 仅用于填充参数，不被使用 | 最简单，无行为 | 填充必需但未使用的参数 |
| **Stub** | 预设的固定返回值 | 被动响应 | 模拟数据库查询结果 |
| **Spy** | 记录调用信息 | 可验证行为 | 验证方法是否被调用 |
| **Mock** | 预设行为预期 | 主动验证 | 验证调用次数、参数 |
| **Fake** | 简化的工作实现 | 有真实逻辑 | 内存数据库替代真实数据库 |

### 示例对比

```javascript
// 真实依赖：从数据库获取用户
class UserRepository {
  async getUser(id) {
    return database.query('SELECT * FROM users WHERE id = ?', [id]);
  }
}

// Stub：返回固定值
const stubRepository = {
  getUser: (id) => Promise.resolve({ id, name: 'Test User' })
};

// Spy：记录调用
const spyRepository = {
  getUser: jest.fn().mockResolvedValue({ id: 1, name: 'Test User' })
};
// 后可以验证：expect(spyRepository.getUser).toHaveBeenCalledWith(1);

// Mock：验证行为预期
const mockRepository = {
  getUser: jest.fn()
    .mockResolvedValueOnce({ id: 1, name: 'User 1' })
    .mockRejectedValueOnce(new Error('Not found'))
};

// Fake：简化的内存实现
class FakeUserRepository {
  constructor() {
    this.users = new Map([[1, { id: 1, name: 'User 1' }]]);
  }
  async getUser(id) {
    return this.users.get(id);
  }
  async save(user) {
    this.users.set(user.id, user);
  }
}
```

> **对你的启发**：Mock 不是越多越好。过度 Mock 会导致测试与实现细节耦合，一改代码测试就挂。优先使用 Fake 或真实依赖（集成测试）。

## 代码覆盖率：度量的艺术

### 四种常见覆盖率

#### 语句覆盖

测量有多少代码语句被执行。

```
function example(a, b) {
  if (a > 0) {        // 语句 1
    return a + b;     // 语句 2
  }
  return b;           // 语句 3
}

// 测试：example(1, 1)
// 执行语句 1、2，覆盖率 = 66.7%
```

#### 分支覆盖

测量每个判断的真假分支是否都被执行。

```
function example(a, b) {
  if (a > 0) {        // 分支 A：true / false
    return a + b;
  }
  return b;
}

// 测试：example(1, 1) 走 true 分支
// 测试：example(-1, 1) 走 false 分支
// 分支覆盖率 = 100%
```

#### 函数覆盖

测量有多少函数被调用。

#### 行覆盖

测量有多少行代码被执行（类似语句覆盖，但按行计算）。

### 覆盖率的关系

```
路径覆盖（最难，组合爆炸）
    ↑
分支覆盖
    ↑
语句覆盖（最基础）
```

**覆盖能力递增**：语句 < 分支 < 路径

### 100% 覆盖率的陷阱

**错误认知**：覆盖率 100% = 没有缺陷

**实际情况**：

```javascript
// 这个测试覆盖率 100%，但毫无意义
describe('meaningless test', () => {
  it('does nothing useful', () => {
    add(2, 3);          // 执行了语句
    add(-1, 1);         // 执行了另一分支
    expect(true).toBe(true);  // 断言永远通过！
  });
});
```

**问题**：
- 测试执行了代码，但没有验证行为
- 断言毫无意义，发现不了任何 Bug

> **对你的启发**：覆盖率是"必要但不充分"指标。目标不是数字，而是**有意义的断言**。定期 Code Review 测试代码，确保测试真的在测东西。

## 常见测试框架速览

| 语言 | 主流框架 | 特点 |
|------|----------|------|
| JavaScript/TypeScript | Jest | 零配置、内置断言、Mock、快照测试 |
| JavaScript/TypeScript | Mocha + Chai | 灵活组合、社区丰富 |
| Python | Pytest | 简洁语法、强大插件生态 |
| Java | JUnit 5 | 标准化、企业级 |

| Rust | 内置 test 宏 | 编译时集成 |

**选择建议**：
- 跟随团队/项目已有选择
- 新项目选主流框架（社区大、资料多）
- 优先选有良好 IDE 集成的框架

## 总结与落地建议

### 决策流程

```
开始写代码
    │
    ▼
需要测试吗？ ──────────────────────────────────┐
    │                                          │
   是                                          │
    │                                          │
    ▼                                          │
是什么类型的代码？                              │
    │                                          │
    ├── 工具函数/纯逻辑 ──▶ 单元测试 + TDD       │
    │                                          │
    ├── API/服务交互 ───▶ 集成测试 + 契约测试    │
    │                                          │
    ├── UI/用户流程 ────▶ E2E 测试（少量关键路径）│
    │                                          │
    └── 发布/重构保障 ──▶ 回归测试套件 + 关键路径自动化      │
                                                │
    ◀───────────────────────────────────────────┘
```

### 核心原则

1. **测试金字塔**：70/20/10 的比例是经验法则，根据项目调整
2. **有意义断言**：测试的目的是发现问题，不是追求数字
3. **TDD 思维**：先想清楚"什么是正确"，再写代码
4. **持续维护**：把测试当作产品资产，随需求演化同步更新用例与替身策略

### 一页速查

```
┌────────────────────────────────────────────────────────────────┐
│                     测试知识速查卡                              │
├────────────────────────────────────────────────────────────────┤
│ 测试层级：单元 → 集成 → E2E → 验收                             │
│ 测试金字塔：大量单元 + 适量集成 + 少量 E2E                       │
│ TDD 循环：Red（写失败测试）→ Green（写通过代码）→ Refactor（重构）│
│ 测试替身：Dummy（占位）Stub（固定返回）Spy（记录）Mock（验证）    │
│           Fake（简化实现）                                     │
│ 覆盖率：语句 < 分支 < 路径，100% 不等于无 Bug                   │
│ 回归：变更后重跑自动化；脆性测试需重构而非一味加断言             │
└────────────────────────────────────────────────────────────────┘
```
