## 快速概览

**一句话结论**：Agent = 大模型 + 记忆 + 工具 + 规划 + 行动；从「只能说」到「能做事」，核心变量是 **Workflow（流程编排）** 和 **Context（信息环境）**。选型时先看场景落在四象限哪一类，再从简单工作流向多 Agent 演进。

## 概述

**AI Agent（智能体）** 是能够感知环境、做出决策并执行行动的智能系统。与传统 AI 应用（输入→处理→输出）不同，Agent 具有 **感知→思考→决策→行动→反馈** 的完整循环。

**核心公式**：

```
Agent = LLM（决策核心）+ Planning（规划）+ Memory（记忆）+ Tools（工具使用）
```

**为什么重要**：

- 从"对话（Chat）"转向"行动（Action）"，解决大模型的"静态知识"和"幻觉"局限
- 2026 年预计企业应用集成 AI Agent 的比例将从 5% 上升到 40%（Gartner 预测）
- 正在重塑软件交互范式，从 Copilot（辅助）走向 Autopilot（自主）

**学习后能做什么**：

- 理解 Agent 系统的设计原理和架构选型
- 使用主流框架（LangChain、LangGraph）构建 Agent 应用
- 评估和优化 Agent 系统的性能和稳定性

## 核心概念

### 1. Agent 的本质定义

**定义**：Agent 是能够感知环境并通过行动影响环境的智能系统，具有完整的"感知→思考→决策→行动→反馈"循环。

**关键点**：

- **LLM（大模型）** = 大脑（具备知识与推理能力，但封闭）
- **Agent（智能体）** = 大脑 + 手脚（Tools）+ 记忆（Memory）+ 感官（Sensors）
- **核心差异**：Chatbot 止步于"输出文本"，Agent 旨在"完成任务"

**示例对比**：

```
用户："帮我分析上季度销售下滑的原因"

Chatbot：输出一段分析文本（可能是编造的）
Agent：读取数据文件 → 编写 Python 代码 → 执行分析 → 生成图表 → 输出结论
```

> [!tip] 延伸阅读
> 详细的 Agent 与 LLM、Chatbot、Tool、Skill 等概念的对比，见 [AI 核心术语解析](#ai-核心术语解析) 章节

---

### 2. 核心变量：Workflow 与 Context

**定义**：Agent 系统的核心由两个关键变量决定——**Workflow（工作流）** 和 **Context（上下文）**。

**关键点**：


| 变量           | 定义            | 特征                       | 来源                    |
| ------------ | ------------- | ------------------------ | --------------------- |
| **Workflow** | 控制任务走向的流程编排模式 | 将复杂任务拆解为有序步骤，明确调用关系和条件分支 | 设计时定义 / 动态生成          |
| **Context**  | 控制内容生成的信息环境   | 包含任务相关的背景信息、知识库、历史记录     | 外部检索、知识图谱、对话历史、工具调用结果 |


**核心洞察**：

> [!info] 核心洞察
> **自动化解决"确定性"问题，智能化解决"不确定性"问题。**

**示例**：

- **Workflow 确定**：发票处理流程（扫描→OCR→验证→入库）
- **Workflow 不确定**：市场调研（需要自主规划路径）
- **Context 确定**：结构化的订单数据
- **Context 不确定**：用户的自然语言问题

---

### 3. Agent 的四层架构

**定义**：Agent 系统由四个核心层组成。

**关键点**：


| 层次                  | 功能      | 组件                                       |
| ------------------- | ------- | ---------------------------------------- |
| **感知层（Perception）** | 接收和理解输入 | 文本输入、多模态输入、环境状态                          |
| **认知层（Cognition）**  | 推理和规划   | System 1（快思考/直觉）*、System 2（慢思考/推理）*、规划模块 |
| **行动层（Action）**     | 执行操作    | 代码工具、检索工具、API 工具、GUI 工具                  |
| **记忆层（Memory）**     | 存储和检索   | 工作记忆、长期记忆（RAG）                           |


**示例**：

```
感知层：接收用户问题 "分析销售数据"
认知层：规划步骤（读取数据 → 分析 → 可视化 → 总结）
行动层：调用 Python 工具执行代码
记忆层：存储分析结果，供后续引用
```

> [!info] System 1 vs System 2
> 源自认知心理学家 Daniel Kahneman 的理论。**System 1** 是快速、自动、直觉式的思维（如识别人脸）；**System 2** 是缓慢、刻意、分析式的推理（如复杂计算）。在 Agent 中，简单任务直接响应（System 1），复杂任务需要多步推理（System 2）。

> 以上概念的详细解释、对比和关系图，见下一章节 [AI 核心术语解析](#ai-核心术语解析)

### 1. LLM（Large Language Model，大语言模型）

**定义**：基于 Transformer 架构、通过海量文本数据训练的大规模神经网络模型，具备文本理解、生成、推理能力。

**与 Agent 的关系**：

- **LLM 是 Agent 的"大脑"**，提供推理和决策能力
- Agent 是 LLM 的"增强版"，为 LLM 加上了手脚（Tools）、记忆（Memory）、感官（Sensors）

**类比**：


| 概念    | 类比   | 能力                |
| ----- | ---- | ----------------- |
| LLM   | 大脑   | 思考、推理、生成文本        |
| Agent | 完整的人 | 思考 + 行动 + 记忆 + 感知 |


**代表产品**：GPT-4、Claude、Gemini、Llama、Qwen

**LLM vs Agent vs Chatbot 的区别**：


| 维度       | LLM       | Chatbot    | Agent              |
| -------- | --------- | ---------- | ------------------ |
| **本质**   | 语言模型（引擎）  | 对话产品（界面）   | 智能系统（架构）           |
| **输出**   | 文本回复      | 文本回复       | 文本 + **实际操作**      |
| **记忆**   | 无（每次调用独立） | 会话内记忆      | 短期 + 长期记忆          |
| **工具使用** | 无         | 通常无        | 可调用外部工具            |
| **示例**   | GPT-4 模型  | ChatGPT 产品 | Cursor Agent、Devin |


---

### 2. Context（上下文）

**定义**：输入给 LLM 的所有信息的总和，包括系统提示、用户输入、对话历史、检索结果、工具调用结果等。

**与 Agent 的关系**：

- **Context 是 Agent 的"信息环境"**，决定了 Agent 能"看到"什么
- Agent 的聪明程度很大程度上取决于 Context 的质量和丰富度

**Context 的组成**：

```
Context = System Prompt（系统提示）
        + User Input（用户输入）
        + Conversation History（对话历史）
        + Retrieved Knowledge（检索的知识）
        + Tool Results（工具调用结果）
        + Memory（记忆内容）
```

**关键概念**：

- **Context Window**：LLM 能处理的最大 Token 数量（如 128K、200K）
- **Context Engineering**：管理和优化上下文的工程实践
- **RAG**：通过检索动态补充 Context

**Context vs Memory 的区别**：


| 维度       | Context（上下文）       | Memory（记忆）      |
| -------- | ------------------ | --------------- |
| **生命周期** | 单次请求内              | 跨请求持久化          |
| **存储位置** | LLM 输入窗口内          | 外部存储（向量库、数据库）   |
| **容量限制** | 受 Token 限制（如 128K） | 理论上无限           |
| **访问方式** | 直接输入给 LLM          | 需要检索后放入 Context |


> [!info] 关系
> Memory 中的内容需要检索后放入 Context，LLM 才能"看到"。

### 3. Agent（智能体）

> [!info] 本节范围
> 基本定义见 [Agent 的本质定义](#1-agent-的本质定义)，本节重点阐述 Agent 与其他概念的关系。

**核心公式**：

```
Agent = LLM（决策核心）+ Planning（规划）+ Memory（记忆）+ Tools（工具）
```

**Agent 如何整合各概念**：


| 概念       | Agent 如何使用它    | 类比      |
| -------- | -------------- | ------- |
| LLM      | 作为决策核心，负责推理和规划 | 大脑      |
| Context  | 作为信息环境，提供决策依据  | 视野/感知范围 |
| Prompt   | 作为指令输入，定义任务和行为 | 任务说明书   |
| Tools    | 作为执行能力，完成实际操作  | 手脚      |
| Memory   | 作为记忆系统，存储和检索信息 | 笔记本/档案  |
| Workflow | 作为流程框架，编排执行顺序  | 工作流程图   |
| Skill    | 作为能力模块，封装特定功能  | 专业技能证书  |


---

### 4. Prompt（提示词）

**定义**：输入给 LLM 的文本指令，用于引导模型生成期望的输出。

**与 Agent 的关系**：

- **Prompt 是与 LLM 交互的"语言"**
- Agent 系统中，Prompt 定义了 Agent 的角色、行为规则、任务目标

**Prompt 的类型**：


| 类型                   | 作用                | 示例              |
| -------------------- | ----------------- | --------------- |
| **System Prompt**    | 定义 Agent 的角色和行为规则 | "你是一个代码审查专家..." |
| **User Prompt**      | 用户的具体任务请求         | "分析这段代码的性能问题"   |
| **Few-shot Prompt**  | 提供示例引导输出格式        | "示例输入→示例输出"     |
| **Chain-of-Thought** | 引导逐步推理            | "让我们一步一步思考..."  |


**Prompt vs Context 的区别**：


| 维度  | Prompt                | Context          |
| --- | --------------------- | ---------------- |
| 范围  | Prompt 是 Context 的一部分 | Context 包含所有输入信息 |
| 性质  | 主要是指令和引导              | 包含指令、数据、历史等      |
| 来源  | 人工设计                  | 人工设计 + 动态生成      |


---

### 5. Workflow（工作流）

**定义**：控制任务执行顺序和流程的编排模式，将复杂任务拆解为有序的步骤序列。

**与 Agent 的关系**：

- **Workflow 定义了 Agent "怎么做"**
- 是 Agent 系统中控制任务走向的核心变量之一

**Workflow 的模式**：


| 模式              | 特点              | 示例             |
| --------------- | --------------- | -------------- |
| **Chain（链式）**   | A→B→C，线性执行      | 写大纲→写正文→校对     |
| **Routing（路由）** | 根据条件分流          | 意图识别→分发到不同处理流程 |
| **Graph（图）**    | DAG* 状态机，支持分支循环 | 复杂审批流程         |
| **Dynamic（动态）** | Agent 自主规划路径    | 开放式任务探索        |


> [!info] DAG（Directed Acyclic Graph，有向无环图）
> 一种图数据结构，节点之间有方向性连接且不形成闭环。在 Workflow 中，DAG 用于表示任务依赖关系：每个节点是一个步骤，边表示执行顺序，"无环"保证任务不会无限循环。

**Workflow vs Agent 的区别**：


| 维度   | Workflow   | Agent       |
| ---- | ---------- | ----------- |
| 确定性  | 通常预定义，确定性高 | 可以动态决策，确定性低 |
| 灵活性  | 按固定流程执行    | 可以根据情况调整    |
| 适用场景 | 流程清晰的任务    | 需要自主探索的任务   |


---

### 6. Function Calling（函数调用）

**定义**：LLM 识别用户意图后，生成结构化的函数调用请求（包含函数名和参数），由外部系统执行并返回结果。

**与 Agent 的关系**：

- **Function Calling 是 Agent 使用工具的底层机制**
- 让 LLM 从"只能说"变成"能做事"

**工作流程**：

```
用户请求 → LLM 分析意图 → 生成函数调用 JSON → 外部执行 → 返回结果 → LLM 整合回复
```

**示例**：

```json
// 用户："北京今天天气怎么样？"
// LLM 生成的 Function Call：
{
  "name": "get_weather",
  "arguments": {
    "location": "北京",
    "date": "today"
  }
}
```

**Function Calling vs Tools 的区别**：


| 维度  | Function Calling | Tools                  |
| --- | ---------------- | ---------------------- |
| 层次  | 底层机制             | 上层抽象                   |
| 定义  | LLM 的原生能力        | 对 Function Calling 的封装 |
| 使用  | 需要定义 JSON Schema | 框架提供更友好的 API           |


---

### 7. MCP（Model Context Protocol，模型上下文协议）

**定义**：Anthropic 开发的开放协议，标准化 LLM 访问外部数据、工具、提示的方式。

**与 Agent 的关系**：

- **MCP 是 Agent 连接外部世界的"标准接口"**
- 解决了 N×M 集成问题（每个 LLM vs 每个工具单独适配）

**类比**：


| 类比       | MCP 的作用   |
| -------- | --------- |
| USB-C 接口 | 统一的物理连接标准 |
| HTTP 协议  | 统一的数据传输标准 |


**MCP vs Function Calling 的区别**：


| 维度  | Function Calling | MCP               |
| --- | ---------------- | ----------------- |
| 层次  | LLM 的原生能力        | 工具连接的协议标准         |
| 范围  | 单次函数调用           | 工具发现、调用、资源访问的完整协议 |
| 标准化 | 各厂商实现不同          | 统一的开放标准           |
| 生态  | 绑定特定 LLM         | 跨 LLM、跨平台         |


**MCP 的组成**：

- **Tools**：可调用的函数/工具
- **Resources**：可访问的数据资源（文件、数据库等）
- **Prompts**：预定义的提示模板

---

### 8. Skill（技能）

**定义**：Agent 的能力模块，封装了特定领域的知识、工具、工作流，使 Agent 能够完成特定类型的任务，其核心价值在于让领域经验成为一种可复用、可迭代和可组合的数字资产。

**与 Agent 的关系**：

- **Skill 是 Agent 的"专业技能包"**
- 一个 Agent 可以具备多个 Skill，实现能力组合

**Skill 的组成**：

```
Skill = 领域知识（Prompt/Instructions）
      + 专用工具（Tools）
      + 工作流模板（Workflow）
      + 资源引用（Resources）
```

**Skill vs Tool 的区别**：


| 维度  | Tool            | Skill            |
| --- | --------------- | ---------------- |
| 粒度  | 单一功能            | 功能集合             |
| 复杂度 | 执行一个操作          | 完成一类任务           |
| 组成  | 一个函数            | 知识 + 工具 + 流程     |
| 示例  | `get_weather()` | "天气助手"（查询+分析+建议） |


**Skill vs Agent 的区别**：


| 维度  | Skill       | Agent                       |
| --- | ----------- | --------------------------- |
| 自主性 | 被 Agent 调用  | 自主决策和行动                     |
| 完整性 | 能力模块        | 完整的智能系统                     |
| 关系  | Agent 的组成部分 | 由 LLM + Skills + Memory 等组成 |


**Skill vs MCP 的区别**：


| 维度       | Skill          | MCP                 |
| -------- | -------------- | ------------------- |
| **层次**   | 应用层（能力封装）      | 协议层（连接标准）           |
| **关注点**  | "做什么"——封装领域能力  | "怎么连"——标准化工具接入      |
| **包含内容** | 知识 + 工具 + 流程   | 工具 + 资源 + 提示模板      |
| **复用方式** | 作为 Agent 的能力模块 | 作为工具的连接协议           |
| **示例**   | "文档写作 Skill"   | MCP Server 提供文件读写工具 |


> [!info] 关系
> Skill 可以通过 MCP 协议接入外部工具，MCP 是 Skill 获取工具能力的标准化通道。

---

> **LLM** 是大脑，通过 **Prompt** 接收指令，在 **Context** 中理解任务，使用 **Function Calling** 机制调用 **Tools**（通过 **MCP** 协议标准化），组合成 **Skills** 形成专业能力，按照 **Workflow** 编排执行，最终构成完整的 **Agent** 系统。

### 1. 场景分类模型（四象限法则）

**理论要点**：用 Workflow 与 Context 的确定性组合划分场景，指导「先做哪类应用、用哪种策略」。

基于 Workflow 和 Context 的确定性组合，将 Agent 应用场景分为四类：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Agent 应用场景四象限分类                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                        Context 确定              Context 不确定             │
│                            ▼                          ▼                     │
│         ┌─────────────────────────────┬─────────────────────────────┐      │
│         │                             │                             │      │
│         │   【象限一】RPA 自动化      │   【象限二】客服/Copilot    │      │
│         │                             │                             │      │
│ Workflow│   • 发票处理、表单填报      │   • 智能客服、代码补全      │      │
│ 确定    │   • 数据迁移、批量操作      │   • FAQ 问答、文档助手      │      │
│    ▶    │                             │                             │      │
│         │   策略：规则驱动 + AI 辅助  │   策略：固定工作流 + RAG    │      │
│         │   重点：稳定性、可预测性    │   重点：语义理解、检索质量  │      │
│         │                             │                             │      │
│         ├─────────────────────────────┼─────────────────────────────┤      │
│         │                             │                             │      │
│         │   【象限三】自主调研        │   【象限四】通用智能体      │      │
│         │                             │                             │      │
│ Workflow│   • 市场分析、代码生成      │   • 创新方案设计            │      │
│ 不确定  │   • 竞品研究、数据探索      │   • 复杂问题解决            │      │
│    ▶    │                             │   • 开放式任务              │      │
│         │   策略：强化学习/自主规划   │   策略：多 Agent + 丰富工具 │      │
│         │   重点：规划能力、路径探索  │   重点：防幻觉、防循环      │      │
│         │                             │                             │      │
│         └─────────────────────────────┴─────────────────────────────┘      │
│                                                                             │
│   复杂度：象限一 < 象限二 < 象限三 < 象限四                                 │
│   建议：从象限一/二开始，逐步向象限三/四演进                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**读图结论**：① 先判断你的场景是 Workflow 确定/不确定 × Context 确定/不确定，落在哪一象限；② 象限一/二优先规则驱动或固定工作流+RAG，象限三/四才需要自主规划或多 Agent；③ 落地时从低象限起步，再按需演进到高象限。

**设计原则**：

- **象限一**：AI 是"粘合剂"，强调稳定性和可预测性
- **象限二**：需要语义理解和信息补全，依赖 RAG 检索质量
- **象限三**：Agent 需要自主规划路径，重点是规划能力
- **象限四**：最复杂，需要防止"幻觉"和无限循环，通常需要多 Agent 协作

### 2. 经典认知架构范式

**理论要点**：Agent 的「思考方式」可归纳为三种范式——ReAct（边想边做）、Plan-and-Solve（先规划后执行）、Reflection（执行后评估再修正）；选型时按任务确定性、灵活性需求权衡。

#### 三种范式流程对比图

│  │  │Thought│───▶│Action│───▶│ Obs  │───▶│Thought│───▶│Action│─...   │    │

**读图结论**：① ReAct 边想边做、适合需要实时反馈的场景（如调试、问答）；Plan-and-Solve 先规划再执行、适合固定流程；Reflection 执行后评估再修正、适合要高质量输出的场景。② 选型时按「确定性 vs 灵活性」权衡：ReAct 最灵活但可能循环，Plan-and-Solve 最稳定但难调整。

#### 三种范式对比表


| 维度       | ReAct     | Plan-and-Solve | Reflection |
| -------- | --------- | -------------- | ---------- |
| **核心思想** | 边想边做      | 三思而后行          | 自我批判修正     |
| **确定性**  | 低（动态调整）   | 高（按计划执行）       | 中（迭代改进）    |
| **灵活性**  | 高         | 低              | 中          |
| **适用场景** | 代码调试、实时问答 | 文档生成、固定流程      | 代码审查、高质量内容 |
| **风险**   | 可能陷入循环    | 计划失效难调整        | 评估标准难定义    |


---

#### 2.1 ReAct（Reasoning + Acting）

**流程**：思考（Thought）→ 行动（Action）→ 观察（Observation）→ 再思考 → ...

**特点**：

- 边想边做，动态调整
- 根据行动结果调整推理方向
- 适合需要实时反馈的场景

**适用场景**：代码调试、数据分析、实时问答

**优势**：灵活，能根据执行结果调整策略  
**劣势**：可能陷入循环，需设置最大迭代次数

---

#### 2.2 Plan-and-Solve（规划后执行）

**流程**：规划（Planning）→ 执行步骤 1 → 执行步骤 2 → ... → 完成

**特点**：

- "三思而后行"，先生成完整计划
- 执行效率高，可预测性强
- 灵活性较低

**适用场景**：文档生成、代码重构、固定流程任务

---

#### 2.3 Reflection（反思机制）

**流程**：执行 → 评估结果 → 识别问题 → 修正 → 再执行 → ...

**特点**：

- 赋予"自我批判和修正"能力
- 能够自我纠错，提升输出质量
- 需要明确的评估标准

**适用场景**：代码审查、架构设计、高质量内容生成

---

### 3. 编排模式

**理论要点**：Workflow 的形态决定步骤如何串联——链式、路由、图（DAG）分别对应线性、分流、分支/循环；与认知范式结合可指导具体框架选用（如 LangGraph 用图）。

| 模式              | 定义             | 特点           | 典型应用            |
| --------------- | -------------- | ------------ | --------------- |
| **Chain（链式）**   | A → B → C，线性执行 | 流程清晰，易于调试    | 写大纲 → 写正文 → 翻译  |
| **Routing（路由）** | 意图识别分流         | 专业化，成本可控     | 客服系统：售后/技术/咨询分流 |
| **Graph（图）**    | 基于 DAG 的状态机    | 支持条件分支、循环、并行 | LangGraph 复杂工作流 |


---

### 4. MCP（Model Context Protocol）协议

**理论要点**：工具与模型的连接需要标准化协议，以降低 N×M 集成成本；MCP 是当前主流选择，与「工具层」设计直接相关。

**定义**：Anthropic 开发的开放协议，标准化 LLM 访问外部数据、工具、提示的方式。

**关键点**：

- **类比**：AI 的"USB-C 接口"，实现即插即用
- **解决问题**：N×M 集成问题（每个 LLM vs 每个工具需要单独适配）
- **状态**：2024 年推出，2026 年已捐赠给 Linux Foundation 下的 AAIF

**技术规范**：


| 组件         | 详情                                      |
| ---------- | --------------------------------------- |
| **传输层**    | HTTP（Streamable HTTP）、SSE、stdio         |
| **RPC 机制** | 基于 JSON-RPC 2.0                         |
| **认证**     | OAuth Bearer tokens                     |
| **工具发现**   | 服务器暴露 tools、resources、prompts           |
| **SDK 支持** | Python、TypeScript、Java、C#、Go、Kotlin、PHP |


**生态系统**：

- Claude 原生支持 MCP
- Microsoft 365 集成（Outlook、Teams、OneDrive）
- 社区服务器：文件系统、数据库、GitHub 等

---

### 5. A2UI（Agent-to-User Interface）

**理论要点**：Agent 输出不仅要有「内容」，还要有「呈现方式」的规范；A2UI 解决界面标准化与安全渲染，与 MCP（数据/工具层）互补。

**定义**：Google 发布的开源项目，解决 Agent 交互"效率低"和"渲染难"的问题。

**关键点**：

- **传统方式**：Agent 发送 HTML → iframe 渲染（风格割裂，安全风险）
- **A2UI 方式**：Agent 发送结构化 JSON "蓝图" → 客户端原生渲染

**与 MCP 的区别**：


| 协议       | 解决的问题          | 类比   |
| -------- | -------------- | ---- |
| **MCP**  | "能不能连通"的物理连接问题 | 数据管道 |
| **A2UI** | 标准化模型输出界面的格式   | 交互层  |


**优势**：

- 风格统一，原生体验
- 安全性高（只传数据，不传代码）
- 适合企业级和多 Agent 网络场景

---

### 6. Agent 构建方案对比

**理论要点**：选型时按「代码 vs 低代码 vs 云」「单 Agent vs 多 Agent」「是否需要 Human-in-the-loop」对齐到四象限与认知范式，再选具体框架或 SDK。

#### 6.1 代码驱动方案


| 方案                | 特点                              | 适用场景            | 状态（2026）   |
| ----------------- | ------------------------------- | --------------- | ---------- |
| **LangChain 1.0** | Python 生态最流行，`create_agent` API | 快速原型、中小型项目      | 生产就绪       |
| **LangGraph 1.0** | 图状态机，持久化状态，Human-in-the-loop    | 复杂工作流、多 Agent   | 生产就绪       |

### 8. Agent 评测方法

**理论要点**：Agent 能力需多维度度量（规划、工具使用、记忆、鲁棒性、安全、效率）；评测设计应防过拟合、覆盖真实环境与细粒度失败环节。

#### 8.1 评测维度


| 维度        | 评测内容                    | 重要性    |
| --------- | ----------------------- | ------ |
| **规划与推理** | 长期目标分解、障碍处理、自适应规划       | 核心能力   |
| **工具使用**  | 正确选择和调用外部工具             | 实际任务必需 |
| **记忆与反思** | 记住历史、评估输出、纠正错误          | 在线学习必需 |
| **鲁棒性**   | 输入变化下的一致性、错误恢复          | 生产部署必需 |
| **安全与对齐** | 避免有害输出、隐私保护、抗 prompt 注入 | 合规必需   |
| **效率与成本** | 延迟、Token 消耗、基础设施开销      | 规模化必需  |


#### 8.2 代表性评测框架


| 框架                       | 特点                                             |
| ------------------------ | ---------------------------------------------- |
| **CLASSic**              | 企业级评测：Cost、Latency、Accuracy、Stability、Security |
| **OSWorld / OSUniverse** | 桌面任务评测：GUI 操作、多步骤工作流                           |
| **Humanity's Last Exam** | 多学科基准：2500+ 问题，防止过拟合                           |


#### 8.3 评测最佳实践

1. **分层多维度评测**：单独能力 + 整体任务
2. **真实多样环境**：Web 导航、GUI、多模态、跨领域
3. **持续更新 + 私有测试集**：防止过拟合
4. **细粒度评分**：部分正确得分，识别失败环节
5. **安全与对齐测试**：对抗测试、隐私检查
6. **效率指标**：延迟、资源消耗、成本

### 9. 安全与治理

**理论要点**：自主行动带来 Prompt 注入、工具滥用、幻觉、循环等风险；防护需分层（输入/执行/输出/治理），并与最小权限、可观测、可回滚等治理原则结合。

Agent 系统由于具备自主行动能力，安全风险比传统 AI 应用更高。

#### 9.1 主要安全风险


| 风险类型          | 描述                          | 影响        |
| ------------- | --------------------------- | --------- |
| **Prompt 注入** | 恶意输入诱导 Agent 执行非预期操作        | 数据泄露、权限越界 |
| **工具滥用**      | Agent 被诱导调用敏感工具（如删除文件、发送邮件） | 系统破坏、信息泄露 |
| **幻觉风险**      | Agent 生成虚假信息并基于此采取行动        | 决策错误、信任损失 |
| **无限循环**      | Agent 陷入无效的行动-观察循环          | 资源耗尽、成本失控 |
| **数据隐私**      | Agent 处理敏感数据时的泄露风险          | 合规违规、法律风险 |


#### 9.2 安全防护策略

```
┌─────────────────────────────────────────────────────────────────┐
│                     Agent 安全防护层次                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【输入层防护】                                                  │
│  ├── 输入验证：过滤危险字符、检测注入模式                       │
│  ├── 意图分类：识别恶意请求                                     │
│  └── Rate Limiting：限制请求频率                                │
│                                                                 │
│  【执行层防护】                                                  │
│  ├── 沙箱环境：隔离工具执行环境                                 │
│  ├── 权限最小化：按需授予最小工具权限                           │
│  ├── 操作审批：敏感操作需人工确认（Human-in-the-loop）          │
│  └── 迭代限制：设置最大循环次数                                 │
│                                                                 │
│  【输出层防护】                                                  │
│  ├── 内容过滤：检测有害/敏感输出                                │
│  ├── 结果验证：使用工具验证 Agent 输出                          │
│  └── 幻觉检测：交叉验证关键事实                                 │
│                                                                 │
│  【治理层】                                                      │
│  ├── 审计日志：记录所有决策和操作（append-only）                │
│  ├── 可观测性：追踪执行路径、状态转换                           │
│  └── 回滚机制：支持操作撤销和状态恢复                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 9.3 治理最佳实践


| 实践         | 说明                           |
| ---------- | ---------------------------- |
| **最小权限原则** | 只授予 Agent 完成任务所需的最小工具和数据访问权限 |
| **分层审批**   | 低风险操作自动执行，高风险操作需人工审批         |
| **操作可逆**   | 优先选择可撤销的操作，避免不可逆变更           |
| **日志完整**   | 记录完整的输入、推理过程、工具调用、输出         |
| **定期审计**   | 定期审查 Agent 行为模式，识别异常         |
| **版本控制**   | Prompt、工具、Workflow 均需版本管理    |

### 示例一：RAG 文档问答 Agent

**目的**：在保持结构清晰的前提下，实现一个**有实际用途**的 Agent：用户对本地文档库提问，Agent 先检索相关片段再生成回答。对应本文 [四象限](#1-场景分类模型四象限法则) 中的「象限二」（Workflow 相对确定 + Context 通过 RAG 补全），以及 [Context](#2-context上下文) 与 [RAG](#2-context上下文) 的实践。

**功能**：用户输入自然语言问题（如「报销流程是什么？」），Agent 调用检索工具从文档库取回相关片段，将检索结果注入 Context，再由 LLM 基于检索内容生成答案；支持多轮追问（工作记忆用当轮 messages）。

**系统结构示意**：

```
  用户输入 "报销流程是什么？"
        │
        ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  感知层   │ 接收 user message，解析为 state.messages                        │
  │  认知层   │ LLM 决定：需要调用 search_docs(query="报销流程")                 │
  │  行动层   │ 执行工具 search_docs，返回 Top-K 相关片段                       │
  │  记忆层   │ 将 tool call + 检索结果追加到 state（供下一轮 LLM 使用）         │
  │  认知层   │ LLM 根据「问题 + 检索结果」生成最终回答                          │
  └─────────────────────────────────────────────────────────────────────────┘
        │
        ▼
  输出 "根据文档，报销流程为：1. 提交申请 2. 主管审批 …"
```

**与本文概念的对应**：

| 组成部分 | 在本示例中的体现 |
|----------|------------------|
| **State** | `MessagesState`：messages 数组（user/assistant/tool），承载多轮与工具结果 |
| **LLM** | 认知核心：先决定是否调用检索工具，再基于 Context（含检索结果）生成回答 |
| **Tool** | `search_docs(query)`：对文档库做语义检索（如向量相似度），返回若干片段 |
| **Context** | 用户问题 + 对话历史 + **检索到的文档片段**（RAG 补全，体现「Context 为王」） |
| **Workflow** | ReAct：LLM → 若需检索则 tool_calls → 执行并写回 state → 再调 LLM 生成文本 |
| **场景象限** | 象限二：固定「检索→生成」流程 + 不确定的用户问法，依赖 RAG 质量 |

**代码骨架（LangGraph + 本地向量检索思路）**：

```python
from langgraph.prebuilt import create_react_agent

def search_docs(query: str, top_k: int = 3) -> str:
    """从文档库中检索与 query 最相关的片段。实际实现可接向量库（如 Chroma、Faiss）。"""
    # 示例：模拟检索结果（生产环境替换为真实向量检索）
    docs = [
        "报销流程：1. 在 OA 提交报销申请 2. 主管审批 3. 财务复核 4. 打款至工资卡。",
        "差旅报销需保留发票与行程单，单次超过 5000 元需分管领导审批。",
    ]
    # 真实实现：embedding(query) 与文档向量相似度排序，取 top_k
    return "\n\n".join(docs[:top_k])

agent = create_react_agent(
    model="anthropic:claude-3-5-sonnet-latest",
    tools=[search_docs],
)

# 用户提问 → 检索 → 基于检索结果生成答案
result = agent.invoke({
    "messages": [{"role": "user", "content": "公司报销流程是什么？需要哪些材料？"}]
})
print(result["messages"][-1].content)
```

**小结**：该示例既保留 **State、LLM、Tool、Workflow（ReAct）、Context（含 RAG）** 的完整结构，又具备真实场景价值（文档问答）；将 `search_docs` 换成真实向量库即可用于内部知识库、帮助文档等。在此基础上可再扩展多步规划（先拆问题再多次检索）或多工具（检索 + 计算器、查表等）。

### 示例二：OpenClaw 剖析解构

**项目**：[OpenClaw](https://github.com/openclaw/openclaw) —— 自托管的**个人 AI 助手**，支持多端、多通道、多工具，可作为「生产级 Agent 系统」的对照样本。

**一句话定位**：在用户已有渠道（WhatsApp、Telegram、Slack、Discord、iMessage、WebChat 等）上提供统一入口，由本地/自建 Gateway 调度 Agent（Pi 运行时）与工具，实现「一个助手、多端回复」。

#### 整体架构（与本文概念对应）

```
  用户接触的通道（WhatsApp / Telegram / Slack / Discord / … / WebChat）
                              │
                              ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  Gateway（控制平面）                                               │
  │  · 单一 WebSocket 控制面（默认 ws://127.0.0.1:18789）               │
  │  · 会话、配置、Cron、Webhook、Control UI、Canvas 托管               │
  └─────────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
  Pi Agent (RPC)         CLI (openclaw …)      WebChat / macOS / iOS·Android 节点
  （推理 + 工具流）         （发消息、配对等）        （语音、Canvas、相机等）
```

- **感知层**：各 Channel 将消息/事件归一后送入 Gateway，再进入对应 Session。
- **认知层**：由 **Pi** 作为 Agent 运行时（RPC 模式），完成规划与推理；支持多 Session、多 Agent 路由（main / 群组等）。
- **行动层**：Gateway 侧统一管理 **Tools**（browser、bash、canvas、nodes、cron、sessions_* 等），Pi 通过 RPC 调用；设备端能力（相机、录屏、通知）通过 **Nodes**（macOS/iOS/Android）以 `node.invoke` 执行。
- **记忆层**：Session 级对话与状态；Workspace 内 `AGENTS.md`、`SOUL.md`、`TOOLS.md` 等注入为上下文；Skills 提供可插拔能力与知识。

#### 核心子系统简表

| 子系统 | 作用 | 对应本文概念 |
|--------|------|--------------|
| **Gateway** | 单一控制平面：会话、通道、配置、Cron、Webhook、Control UI | Workflow 编排、Context 来源、安全与策略 |
| **Pi Agent** | 实际跑推理与工具调用的 Agent 运行时（RPC） | LLM + Planning + Tools 执行 |
| **Channels** | WhatsApp / Telegram / Slack / Discord / Google Chat / Signal / iMessage / BlueBubbles / Teams / Matrix / Zalo / WebChat 等 | 感知层输入、行动层输出（多模态收发包） |
| **Tools** | browser、canvas、bash、nodes、cron、sessions_list / sessions_send、webhooks 等 | 行动层工具集；部分等价「Skill」能力 |
| **Skills** | 工作区 `~/.openclaw/workspace/skills/<skill>/SKILL.md`，可安装/管理 | Skill（知识+流程+工具封装） |
| **Nodes** | macOS / iOS / Android 节点，提供相机、录屏、Canvas、通知、位置等 | 行动层在设备端的延伸 |
| **Session 模型** | main / 群组、激活方式、队列、回复策略 | 多 Agent/多会话路由与隔离 |

#### 设计要点（可借鉴处）

1. **本地优先**：Gateway 与 Pi 可完全跑在自有设备/服务器，数据与模型调用可控。
2. **通道与 Agent 解耦**：同一套 Agent 能力复用到十几种通道，仅做通道侧适配（DM 策略、群组规则等）。
3. **多 Agent/多会话**：通过 Session 与 `sessions_*` 工具做 Agent 间协作（列表、历史、发送、spawn），对应「多 Agent 路由与协作」。
4. **安全与沙箱**：非 main 会话可进 Docker 沙箱；DM 默认 pairing 白名单；Gateway 可仅绑定 loopback，通过 Tailscale Serve/Funnel 或 SSH 暴露。
5. **Skills 与注入文件**：`AGENTS.md` / `SOUL.md` / `TOOLS.md` 注入到上下文，Skills 以目录+SKILL.md 形式扩展，便于理解「Context 来源」与「Skill 与 MCP 的互补」（OpenClaw 用自研技能体系，与 MCP 工具协议可并存）。

#### 从理论到实现的映射（深化对照）

| 本文理论/概念 | OpenClaw 中的具体体现 | 说明 |
|---------------|------------------------|------|
| **认知范式**（[ReAct / Plan-and-Solve / Reflection](#2-经典认知架构范式)） | **会话内**：Pi 采用 ReAct 式流式推理（Thought→Action→Observation→…），适合多工具、多步任务。**跨会话**：由 Gateway 与 Session 路由决定「谁处理」——本质是 **Routing**，不是单 Agent 内的 Plan-and-Solve。Reflection 未在默认流水线中显式暴露，但可通过 Skill 或自定义流程做「执行→评估→修正」。 | 同一系统内可分层：单会话用 ReAct，会话间用路由；范式对应到「层级」而非全局唯一。 |
| **Context 组成**（[Context 为王](#2-context上下文)） | **对话历史**（Session 级）+ **注入文件**（`AGENTS.md` / `SOUL.md` / `TOOLS.md` 定义角色、原则、工具说明）+ **Skills**（SKILL.md 提供领域知识与指令）+ **工具返回结果**（browser、bash、sessions_* 等）。Context 的「质量」直接依赖注入文件与 Skill 编写质量。 | 与本文「Context = 系统提示 + 用户输入 + 历史 + 检索/工具结果」完全一致；OpenClaw 把「检索」部分交给注入文件与 Skills，而非独立 RAG 服务。 |
| **四象限**（[场景分类](#1-场景分类模型四象限法则)） | 产品定位偏 **象限二/三**：Workflow 在单会话内由 ReAct 动态决定（偏不确定），Context 通过注入与 Skills 补全（半确定）。多通道、多 Session 则引入「路由确定性」——谁回复、是否进沙箱等由配置与规则决定。 | 理解四象限有助于判断 OpenClaw 适合「个人/团队助手」而非纯 RPA 或纯开放探索。 |
| **编排模式**（[Chain / Routing / Graph](#3-编排模式)） | **会话内**：近似 **Graph**（LangGraph 风格），有分支与工具循环。**跨会话**：**Routing**（按 Channel、Session 类型、DM/群组规则分流）。无显式 DAG 画布，但 Gateway 的会话与队列管理承担了「编排」职责。 | 编排既在 Pi（推理图）也在 Gateway（会话与通道路由）。 |
| **安全与治理**（[安全与治理](#9-安全与治理)） | **输入层**：DM 默认 pairing 白名单，降低匿名滥用。**执行层**：非 main 会话可进 **Docker 沙箱**，限制 bash/filesystem 影响范围；**迭代/循环** 由运行时与配置约束。**治理层**：Gateway 有 **日志、Presence、会话状态**，便于审计与可观测；可仅绑定 loopback，通过 Tailscale/SSH 暴露，控制网络面。 | 与本文「输入/执行/输出/治理」分层防护一致；沙箱 + 最小权限 + 可观测是落地要点。 |

#### 设计取舍分析

| 设计决策 | 好处 | 代价/约束 |
|----------|------|------------|
| **自研 Skills 体系（目录 + SKILL.md）** | 与工作区、版本管理、人类可读格式天然结合；可渐进扩展，不依赖外部协议。 | 与 MCP 生态不直接互通；若需接入大量 MCP 工具，需在 Gateway/Tools 层做桥接。 |
| **Gateway 单一控制平面** | 会话、通道、Cron、Webhook、UI 统一管理；部署简单（一个进程/容器）。 | 单点：Gateway 故障影响所有通道；扩展时需考虑高可用与水平扩展。 |
| **Pi 与 Gateway 分离（RPC）** | 推理与工具执行可独立扩容、升级；Gateway 可对接不同 Agent 实现。 | 网络与序列化开销；调试需同时看 Gateway 与 Pi 日志。 |
| **多 Session / sessions_* 工具** | 支持「多 Agent」协作（列表、发消息、spawn），对应多智能体场景。 | 会话模型与权限边界需在业务上设计清楚，否则易出现越权或混乱。 |

#### 安全与治理在 OpenClaw 中的落地

- **最小权限**：非 main 会话进 Docker 沙箱；DM 仅对已配对用户开放。
- **操作可逆与边界**：bash/文件类工具在沙箱内，降低误删、误改范围；Gateway 可仅监听 loopback，外网通过 Tailscale Serve/Funnel 或 SSH 暴露，控制攻击面。
- **可观测与审计**：Gateway 日志、Presence、会话状态为「谁在何时做了什么」提供基础；可在此基础上加审计日志与告警，对应本文「治理层」的审计与可观测性要求。

**小结**：OpenClaw 把「多通道入口 + 单一控制平面 + 单/多 Session Agent + 丰富工具与 Skills + 设备节点」打成一套可自托管的生产级系统。通过**从理论到实现的映射**可看到：认知范式、Context 组成、四象限、编排模式、安全与治理在文中均有对应，而**设计取舍**（自研 Skills vs MCP、单点 Gateway、Pi 分离、多 Session）则说明真实系统是在约束下做权衡，而非简单套用单一范式。进一步细节见其 [官方文档](https://docs.openclaw.ai) 与 [GitHub](https://github.com/openclaw/openclaw)。
