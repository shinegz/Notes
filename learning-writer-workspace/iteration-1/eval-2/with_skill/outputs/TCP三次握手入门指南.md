# TCP 三次握手入门指南

> 就像打电话前要先拨通确认对方在，三次握手让双方确认"线路通畅、都能说话"。

---

## 30 秒心智模型

| 步骤 | 电话类比 | TCP 动作 |
|-----|--------|---------|
| 第一步 | 拨号 | 客户端发送 SYN |
| 第二步 | 对方说"喂" | 服务器返回 SYN+ACK |
| 第三步 | 你说"你好" | 客户端发送 ACK |

---

## 什么是 TCP

TCP（传输控制协议）是互联网上最重要的协议之一。

**核心特点**：
- **可靠传输**：数据丢了我重发
- **有序到达**：先发的先到
- **面向连接**：通信前先"握手"

---

## 三次握手详解

### 第一次握手：客户端请求

```
客户端                              服务器
  |                                  |
  | -------- SYN Seq=x -------------> | LISTEN
  | 状态: SYN-SENT                   |
```

客户端发送 SYN（同步）报文，包含自己的初始序列号 x。

### 第二次握手：服务器响应

```
客户端                              服务器
  |                                  |
  | <------- SYN+ACK Seq=y --------- | SYN-RECEIVED
  |        Ack=x+1                   |
```

服务器返回 SYN+ACK，表示"收到，我也准备好了"。

### 第三次握手：客户端确认

```
客户端                              服务器
  |                                  |
  | -------- ACK Ack=y+1 ----------> | ESTABLISHED
  | 状态: ESTABLISHED               |
```

客户端确认，双方都准备好，可以传输数据了。

## 完整流程图

```
          (CLOSED)                      (LISTEN)
             |                             |

               连接建立成功！
```

## 常见问题

### Q: 为什么不能是两次握手？

**原因**：无法确认双向都能通信。

如果只有两次握手：
- 客户端说"我要连接"
- 服务器说"好的"
- 但服务器不知道自己的回应客户端能不能收到

### Q: 为什么不能是一次握手？

**原因**：无法确认服务器的接收能力。

一次握手只能证明服务器"能接收"，不能证明它"能发送"。

### Q: SYN 会携带数据吗？

**答案**：不会。SYN 和 ACK 报文不携带应用数据，只消耗序列号。

---

## 总结

1. **第一次握手**：客户端发送 SYN，证明自己能发
2. **第二次握手**：服务器返回 SYN+ACK，证明自己能收
3. **第三次握手**：客户端发送 ACK，确认对方能收自己的回应

三次握手缺一不可，确保了 TCP 连接的可靠性。
