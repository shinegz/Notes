# 学习笔记：架构层（系统论、价值流、服务设计、设计思维）

文献锚点：Meadows (2008)、Rother & Shook (1998)、Stickdorn 等 (2018)、IDEO Design Kit。下面是**接到人机管线**的读法，不是四门课的百科条目。

## 30 秒心智模型

**架构层解决的是：规则有了之后，东西装在哪、信息流怎么跑、坏了从哪截断、你从哪看见里面在发生什么。**  
系统论和价值流帮你找**浪费与瓶颈**；服务设计帮你找**交接面**；设计思维逼你先问「谁怕什么」再画框。四样都服务于同一件事：**可替换、可观测、可回滚**。

## 这层在解决什么

规则定好了，还要**放进结构**：谁和谁交互、信息怎么走、失败怎么扩散或隔离。这一层练的是**切分**：模块边界清楚，自动化才不会长成一坨改不动的脚本。

分工大致这样用：

1. **系统思考（Meadows）**：反馈、存量、杠杆点。她反复强调：**结构先出行为**，光喊「大家要重视」多半没用。[1]  
2. **价值流（Rother & Shook）**：同一张图里看**物料与信息**，用周期、库存、节拍把浪费摊开。[2] 对人机管线：常浪费在重复检索、无效长上下文、人在不该出现的地方出现。  
3. **服务设计（Stickdorn 等）**：旅程、利益相关者、**服务蓝图**——前台动作、后台动作、支持系统画在同一视线里。[3]  
4. **设计思维（IDEO）**：灵感—构思—实现的方法包，强调和用户一起试。[4] 对人机系统，等价问题是：**哪一步允许自动、哪一步必须人点头**，这比「换一个更大的模型」优先。

## 核心概念：再往下一层

### 边界画在哪：模型不是「系统」的全部

「系统」在 Meadows 里是有边界、有反馈环的整体。[1] 对 Agent 管线：模型权重、提示词、工具网关、向量库、人在回路、监控，**谁在边界内、谁是环境**，要画在纸上。边界画错，后面所有「优化模型」都像在屋里换家具却漏了屋顶漏水。

### 价值流：先问排队，再问参数量

价值流图的价值在**数据框**：周期、在制、节拍。[2] 人机类比：

- 用户请求进来以后，**卡在哪一段队列**？（审核？GPU？外部 API？）  
- 信息是**推**还是**拉**？拉式常常更不容易堆垃圾在中间。

瓶颈不在「模型不够大」的时候多得很：瓶颈在**串行审核**或**重复调用**。

### 服务蓝图：用户看见的失败，从哪条线长出来？

蓝图把可见动作与内部动作分开。[3] Agent 场景里至少画清三行：**用户可见输出**、**模型/工具内部**、**支持系统（日志、计费、权限）**。  
很多「模型胡说」在用户眼里是前台事故，根因在后台：**没有失败语义、没有降级路径**。

### 接口与失败语义：比「更聪明的 prompt」便宜

耦合、内聚、契约，老词在工具调用上更值钱。每个工具至少答清四件事：**超时几秒、失败返回什么、会不会重试写两次、部分成功怎么表示**。  
Meadows 说的**延迟与反馈**在这里对上号：观测滞后时，粗暴自动纠偏会振荡——所以要有**分级**（自动 / 告警 / 人）。[1]

### 设计思维：别跳过「共情」直接堆组件

IDEO 类方法的价值在于先弄清**怕什么**（错答、泄密、不可解释、被老板骂）。[4] 架构评审若直接从「我们上一个 RAG」开始，常常少了一步：**失败时用户需要看见什么、能撤销什么**。

## 微型案例：客服草稿管线（骨架）

**当前态（口头价值流）**：用户提问 → 模型起草 → 一线复制粘贴发出 → 若出事，主管人工改口径。  
浪费在哪：起草和发出之间**没有结构化交接**；事故**不可归因**（是模型、是复制的人、还是政策？）。

**未来态（结构改动）**：同一价值流，但在「发出」前加**可观测闸口**：草稿带版本号、敏感操作走二次确认、日志打在**工具边界**而不是散在聊天里。  
这不是换模型，是**把交接面画进蓝图**。[2][3]

## 和 Agent 协作时的落点

| 痛点 | 可动手的地方 |
|------|----------------|
| 上下文爆炸 | 会话态 / 检索 / 长期知识分层；检索策略写成配置，别写死在 prompt |
| 工具滥用 | 网关：配额、审计、风险分级；高危工具强制二次参数 |
| 质量不稳 | 流水线：草稿 → 自动检查 → 人审；灰度与回滚写进发布，不是写进祈祷 |
| 协作扯皮 | RACI：谁定义契约、谁批准变更、谁对指标负责 |

## 易混辨析

- **价值流图 ≠ 流程图**。前者强制你填**时间与库存**。[2]  
- **服务蓝图 ≠ UX 线框**。前者关心**前后台交接与失败**，不只关心按钮。[3]  
- **系统论 ≠ 画很多框**。没有反馈与观测，框再多只是组织结构图。

## 最小练习（任选一个）

1. 七框以内画一条链路，每箭头写：**同步/异步、超时、失败语义**。  
2. 给关键路径画**一行蓝图**：用户可见 / 模型+工具 / 支持系统，标一个「可见失败点」。[3]  
3. 把一段「万能 prompt」拆成检索、推理、格式三段，跑几天看定位故障是否更快。
