# webapp-e2e-testing-v2 设计文档

## 1. 背景

### 1.1 当前状态

`webapp-e2e-testing` skill（现名 `nova-e2e-testing`）已实现完整的工作流程——需求检查、Mock 七步法、静态交互分析、15 条规则索引、Playwright 脚本执行。但它不稳定，同一场景反复失败。

### 1.2 根因诊断

| 问题 | 表现 |
|------|------|
| 食谱书反模式 | 15 条规则按场景索引，Agent 遇到未覆盖场景就失控 |
| 混合抽象层级 | 高层原则（只 mock 网络层）和具体命令（`route.fulfill()`）混在一起，Agent 分不清什么是必须遵守的、什么是可以替换的 |
| 缺少统一测试哲学 | Agent 不理解「测什么」和「以什么标准测」，盲目执行操作清单 |
| 规则爆炸 | 规则从实践失败中增量生长，缺少顶层提炼，数量越多越难内化 |
| 无心智模型 | Agent 死记规则而非理解原理，同样的坑换一个形式就会再踩 |

### 1.3 设计目标

创建一个**纯方法论驱动**的 skill——通过原则清单教 Agent 理解 E2E 测试的本质，而非通过程序化步骤告诉它做什么。Agent 掌握原则后，能自主判断具体操作，适应任意前端项目。

## 2. 核心决策

### 2.1 纯方法论驱动

不包含：具体命令、工具选择、项目结构约定、文件路径。
只包含：原则（title + why + how + anti-pattern）。

Agent 自身将方法论映射到具体工具和项目代码。

### 2.2 原则清单式结构

每条原则独立、可组合，类似 Clean Code / SOLID 风格。Agent 不需要按顺序执行，而是在遇到具体场景时检索相关原则。

每条原则结构：
- **标题**：简洁的原则名称
- **为什么**：解释这条原则解决什么问题
- **如何应用**：Agent 应该怎么做
- **反模式**：常见的错误做法（配示例）

### 2.3 五领域分层（方案 B）

5 个领域按 E2E 测试的自然流程排列，每个领域有前置说明 + 3~5 条原则：

```
测试哲学 → 数据与Mock → 交互策略 → 诊断与调试 → 报告与输出
```

### 2.4 自包含

SKILL.md 包含全部内容，不拆分到外部 references/ 文件。

## 5. 领域设计

### 5.1 测试哲学

**领域说明**：定义「测什么」和「以什么标准测」。Agent 在执行任何测试动作之前，必须先在这些原则上建立锚点。

| # | 原则 | 核心要点 |
|---|------|---------|
| 1 | 先理解，再测试 | 执行前必须明确：用户流程、每步预期状态、通过/失败标准。输入不完整时主动澄清。 |
| 2 | 测试用户可见行为 | 断言验证页面渲染结果而非内部状态。选择器优先语义属性而非实现细节。 |
| 3 | 每次测试独立运行 | 不依赖前次运行残留数据。通过 mock 或 API 创建初始状态。 |
| 4 | 只控制你能控制的 | 第三方服务（支付、外部 API、CDN）用 mock 替代真实请求。 |
| 5 | 关键路径优先 | E2E 聚焦核心用户流程，异常路径留给单元测试。 |

### 5.2 数据与 Mock 策略

**领域说明**：本地 E2E 测试的核心挑战——前端需要数据才能渲染，但后端 API 可能不可用或无法提供目标状态的数据。

| # | 原则 | 核心要点 |
|---|------|---------|
| 1 | 先看代码，再定数据 | Mock 数据前追踪数据从 API 入口到 UI 渲染的完整路径，确认字段类型和转换逻辑。 |
| 2 | Mock 的边界是网络层 | Mock 只能发生在 HTTP 请求拦截层。禁止修改 HTML、替换组件、注入假数据绕过渲染。 |
| 3 | 数据真实性决定测试可靠性 | Mock 字段类型必须与真实 API 一致，数组长度反映真实场景，条件渲染字段设置正确值。 |
| 4 | 状态模拟覆盖所有注入路径 | 三条路径：API 响应（网络拦截）、持久化存储（localStorage/sessionStorage 预填充）、**服务端注入的全局变量**（`window.__xx__`、HTML 内联 `<script>`）。 |

### 5.3 交互策略

**领域说明**：Agent 与 UI 元素交互时的决策框架——不规定「用什么工具」，而是建立「面对一个 UI 元素时应该如何思考」。

| # | 原则 | 核心要点 |
|---|------|---------|
| 1 | 先用语义定位 | 选择器优先级：语义属性 > data-testid > CSS 类名 > DOM 路径。禁止 XPath。 |
| 2 | 操作前先确认元素可达 | 每次交互前确认：DOM 中存在 → 视口内可见 → 未被遮挡 → 未 disabled。 |
| 3 | 理解受控组件的事件模型 | React/Vue 受控组件需要合成事件触发。填值：focus → fill → blur。disabled 按钮可能是合成事件未触发。 |
| 4 | 处理遮挡而非绕过遮挡 | 遇到拦截层先确认是否预期存在。`force=true` 仅限于浮层 iframe 场景。 |
| 5 | 表单提交前诊断 disabled 状态 | 下拉菜单未关闭是最常见原因。排查：截图确认 → 输入值显示 → radio 选中 → 校验错误。 |

### 5.4 诊断与调试

**领域说明**：每次失败都是一次诊断机会，必须找到根因才能修复。Agent 的可靠性取决于面对失败时的反应模式。

| # | 原则 | 核心要点 |
|---|------|---------|
| 1 | 失败可诊断，不可跳过 | 遇到阻塞：确认错了什么 → 追问为什么 → 修复根因重试。禁止换方式碰运气。 |
| 2 | 分层排查，先近后远 | 排查顺序：Console 报错 → 网络请求 → 数据流转换 → 条件渲染 → 交互状态。 |
| 3 | 诊断结论必须有证据支撑 | 结论基于：截图 + 网络日志 + Console 日志 + 源码阅读。四者至少有其二。 |

### 5.5 报告与输出

**领域说明**：测试的最终交付物不是脚本运行成功，而是让用户理解「测了什么、结果如何、如果失败原因是什么」。

| # | 原则 | 核心要点 |
|---|------|---------|
| 1 | 报告回答三个问题 | 测了什么（流程+验证点）→ 怎么测的（关键操作+数据设置）→ 结论可信吗（限制说明）。 |
| 2 | 失败报告的可追溯性优于详细程度 | 包含：失败截图 + 网络状态 + 操作序列 + 已尝试的诊断步骤和结论。 |
| 3 | 截图是核心证据 | 截取完整视口，在关键状态变化点截图，失败截图必须包含实际失败状态。 |

## 6. 与旧版的关系

- 旧版 `webapp-e2e-testing`（名 `nova-e2e-testing`）保持不变
- v2 是完全独立的新 skill，不依赖旧版的 references/
- v2 稳定后可以考虑废弃旧版
